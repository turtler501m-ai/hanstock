# 2. 공통 안전 원칙

기본 운영 설정은 다음과 같이 유지한다.

DRY_RUN=true
TRADING_ENV=demo
ENABLE_LIVE_TRADING=false
REQUIRE_APPROVAL=true

주문 연결 흐름:

AI 종목 발굴
→ 관찰종목 등록
→ 결정적 confirmed 검증
→ 승인된 실행 전략 연결
→ RiskEngine 검사
→ 실행 계획 생성
→ 자동화 정책 평가
→ 기존 승인 대기열
→ 기존 OrderRouter

자동화 레벨:

- 0: 분석만 자동
- 1: 후보 발굴 자동
- 2: 관찰 등록 자동
- 3: confirmed 자동 검증
- 4: 실행 계획 자동 생성
- 5: 승인 대기열 자동 등록
- 6: 주문 자동 실행

기본값:

```text
automation_level=4
auto_approve=false
auto_execute=false
```

Level 5와 Level 6은 전략별 자동화 정책, RiskEngine, 데이터 신선도, 비중·손실·중복 주문 한도 및 실행 환경 안전 조건을 모두 통과한 경우에만 허용한다. 자동 주문도 신규 브로커 우회 경로를 만들지 않고 기존 시장별 승인·주문 경로만 사용한다.

AI 평가 실패 후 안전한 fallback 결과가 없거나 데이터·전략·리스크 검증이 완료되지 않은 경우에는 실행 계획 또는 주문을 생성하지 않는다. stale/fallback 거래 허용 여부는 자동화 정책으로 관리하며 기본값은 모두 `false`다.

demo 환경에서 분석·계획과 자동주문 경로를 먼저 검증한 뒤, 전략 승인·리스크 한도·kill switch 검증 및 사용자 명시 승인을 거쳐서만 실계좌 환경으로 전환한다.

## 목적

## 종목 유니버스

- 한스톡 관심종목

미국:

- 미스톡 관심종목
- 주요 미국주식 목록
- 거래량·유동성 조건 통과 종목
- 내러티브 관련 섹터 종목
- 전략별 전용 유니버스

## 분석 과정

유니버스 구성
→ 데이터 품질 검사
→ 룰 기반 필터
→ 기술 점수
→ 모멘텀 점수
→ 내러티브 점수
→ AI 평가
→ 위험 감점
→ 시장 국면 보정
→ 최종 순위

## 점수 구성

모든 점수는 0~100으로 정규화한다.

base_score =
rule_score × 0.20
+ technical_score × 0.20
+ momentum_score × 0.15
+ narrative_score × 0.25
+ ai_score × 0.20

final_score =
clamp(
base_score
× regime_multiplier
× data_quality_multiplier
- risk_penalty,
0,
100
)

가중치는 전략 profile에서 변경할 수 있게 한다.

## 목록 컬럼

- 순위

- 핵심 근거

## AI 판단

- strong_watch
- watch
- neutral
- avoid
- insufficient_data

AI스톡에서는 `buy`, `sell` 판단을 사용하지 않는다.

## 필수 제한

- 최소 룰 점수 미달 후보는 AI가 승격하지 못한다.
- AI confidence 미달 시 AI 점수의 가중치를 0으로 설정한다.
- AI 오류 시 룰 기반 점수로 fallback한다.
- stale 데이터 후보는 `insufficient_data`로 처리한다.
- AI 후보는 전략 검증, RiskEngine, 승인·자동화 정책을 통과한 뒤 기존 시장별 승인·주문 경로로만 연결한다.

- 시장별 자산 비중
- 종목별 현재 비중
- 종목별 제안 비중
- 섹터별 비중
- 전략별 비중
- 현금 비중
- 포트폴리오 변동성
- 집중도
- 위험 예산 사용률
- 리밸런싱 제안

## 기본 원칙

AI는 목표 비중을 제안할 수 있지만 주문 수량을 확정하지 않는다.

최종 수량은 RiskEngine이 계산한다.

risk_budget = 운용자본 × 거래당 허용 위험률
risk_per_share = entry_price - stop_price
quantity = floor(risk_budget / risk_per_share)

## 제한

- 종목별 최대 비중
- 섹터별 최대 비중
- 전략별 최대 비중
- 시장별 최대 비중
- 동시 보유 수
- 최소 현금 비중
- 일일 신규 투자 한도
- 환율과 통화 구분

한국 원화 자산과 미국 달러 자산을 같은 가격 단위로 직접 합산하지 않는다.

## 안전 정책

성과가 기준 이하이면 자동 재학습하거나 자동 폐기하지 않는다.

전략 상태를 `review_required`로 변경하고 운영자 검토를 요구한다.

# 11. 세부 탭 8: AI 전략 관리

## 목적

AI스톡의 점수 가중치, 모델, 검증 상태와 버전을 관리한다.

## 표시 항목

- 전략명
- 전략 ID
- 적용 시장
- 상태
- 전략 버전
- 모델
- feature version
- prompt version
- 점수 가중치
- 최소 룰 점수
- 최소 confidence
- 최대 위험 점수
- profile hash
- 마지막 검증일
- 마지막 사용일
- 최근 성과

## 전략 상태

draft
→ static_verified
→ verified
→ backtested
→ approved
→ review_required 또는 retired

## 검증 조건

- 점수 가중치 합계 검증
- 입력·출력 schema 검증
- AI API 검증
- 캐시 및 fallback 검증
- out-of-sample 백테스트
- rule-only 비교
- 최소 표본 수
- 거래비용 반영
- profile hash 일치

전략 변경 시 기존 승인 및 백테스트 검증 상태를 무효화한다.

# 12. 세부 탭 9: 실행 계획

## 목적

관찰종목 중 검증된 후보만 기존 실행 계획으로 안전하게 전달한다.

## 생성 조건

- 관찰 상태가 confirmed
- 승인된 실행 전략 존재
- 종합 점수 기준 통과
- 룰 점수 기준 통과
- 위험 점수 기준 통과
- 데이터가 fresh
- 진입가 존재
- stop_price 존재
- risk_per_share > 0
- 유동성 기준 통과
- 중복 주문 없음
- 일일 손실 한도 미초과
- 종목·섹터·전략 비중 한도 통과

## 표시 항목

- 시장
- 종목
- 연결된 AI 후보
- 연결된 실행 전략
- 제안 행동
- 제안 진입가
- stop_price
- take_profit 참고값
- 위험 예산
- 계산 수량
- 예상 주문 금액
- 주요 근거
- 리스크 검사 결과
- 승인 상태

정책상 자동 주문이 허용된 실행 계획은 기존 승인 대기열과 OrderRouter를 사용해 주문을 실행할 수 있다.
브로커 API를 직접 호출하는 우회 경로는 허용하지 않는다.

# 17. 테스트

외부 API는 모두 mock한다.

최소 테스트:

- 모든 세부 탭 API가 market 필터를 적용함
- 한국·미국 응답 schema가 동일함
- 시장별 종목 코드와 통화가 분리됨
- 시장 국면이 최종 점수에 반영됨
- stale 내러티브가 감점됨
- fading/disappeared가 양수 보너스를 받지 않음
- 단일 출처 내러티브가 감점됨
- 룰 기준 미달 후보가 AI로 승격되지 않음
- AI 오류 시 fallback
- 모든 점수가 0~100 범위임
- 위험 점수가 최종 점수를 낮춤
- stale 후보가 confirmed 상태가 되지 않음
- AI만으로 confirmed 상태가 되지 않음
- 관찰종목 상태 전이가 검증됨
- 손절가 없는 후보는 실행 계획을 생성하지 않음
- AI 후보가 정책·리스크 검증 없이 주문으로 실행되지 않음
- 기본 `automation_level=4`에서는 승인 대기열 등록과 주문이 자동 실행되지 않음
- Level 5에서는 승인 대기열 등록까지만 수행됨
- Level 6에서는 모든 안전 조건을 통과한 경우에만 기존 시장별 주문 경로가 호출됨
- 차단 단계와 사유가 자동화 실행 이력에 저장됨
- 동일 시장 스캔의 중복 실행이 방지됨
- 후보 성과가 시장·전략·버전별로 저장됨
- 기존 한스톡·미스톡·내러티브 테스트가 통과함

# 19. UI 구현 원칙

- 기존 한스톡·미스톡·내러티브 화면을 삭제하지 않는다.
- `AI스톡`을 별도 메인 메뉴로 추가한다.
- 세부 탭은 공통 컴포넌트를 재사용한다.
- 한국과 미국은 상단 시장 필터로 전환한다.
- 상세 분석은 모달 또는 우측 상세 패널로 제공한다.
- 로딩, 오류, 빈 데이터, stale 상태를 구분한다.
- 점수 색상만으로 판단하지 않도록 텍스트 상태를 함께 표시한다.
- 현재 수정된 web 파일의 사용자 변경을 보존한다.
