## 0. 개정 요약

AI스톡을 한스톡·미스톡·내러티브 데이터를 결합하는 **별도 메인 대시보드**로 구현한다.

이번 개정에서는 기존 계획의 다음 문제를 바로잡았다.

- 실제 저장소 구조에 맞게 라우터, DB, 화면 진입 방식을 수정했다.
- 9개 세부 탭마다 목적, 데이터 원천, API, 화면, 완료 조건을 정의했다.
- 미스톡 별도 DB와 한스톡 메인 DB의 경계를 명시했다.
- AI 호출, 내러티브 수집, 점수 계산, 주문 검증의 책임을 분리했다.
- 디자인 효과보다 데이터 신뢰성, 접근성, 안전 상태 표시를 우선했다.
- 9개 탭 전체를 한 번에 구현하지 않고 MVP와 고도화 단계로 나눴다.
- 국내 계열을 **AI한스톡(KR)**, 미국 계열을 **AI미스톡(US)** 으로 명확히 명명하고 데이터·DB·통화·전략 경계를 분리했다(§1.3).
- 9개 탭을 독립 화면이 아니라 하나의 분석 파이프라인으로 연결하는 탭 간 데이터 흐름과 연결 키를 정의했다(§5.10).
- 그래프·표·탐색 사용 편의성(정렬·필터·고정·키보드·반응형)을 §8.6으로 구체화했다.
- 실제 코드를 실측해 AI 평가(`ModelPredictor`/OpenAI)·전략 lifecycle 연결 규약(§5.11)과 스케줄 자동 실행·무중단 운영 규약(§5.12)을 명시하고, probability를 confidence로 노출하지 않도록 매핑을 고정했다.
- 세부 탭은 9개로 고정하고, 스케줄 운영 가시성은 탭3·탭8 서브패널로 흡수했다.
- 자동 승인·자동 주문은 전략별 `automation_level`과 리스크·승인 정책(§1.2, §6.6)을 통과한 경우에만, **기존 시장별 승인·주문 경로로만** 수행한다(브로커 API 직접 우회 금지).
- AI를 **시황 해석·종목 선정·매매 타이밍의 주도자**로 격상하고(§4.5), ETF 포함·소형주 제외 유니버스(§4.6), 월/주/일 시황 파일(§4.7), 1차 배치 점수화 + 2차 실시간 타이밍 2단계(§4.8)를 추가했다.
- 모든 운영 수치·처리 기준을 **하드코딩하지 않고 환경설정에서 변경 가능**하게 설계했다(§1.4, 기존 env-settings 화면 확장).

AI스톡의 핵심 흐름은 다음과 같다.

```text
한스톡·미스톡 시장 데이터
        + 내러티브 데이터
        + 기존 룰 전략 결과
                  ↓
       공통 특징 및 데이터 품질 검사
                  ↓
      결정적 점수 + 선택적 AI 평가
                  ↓
      AI스톡 후보·상세 분석·관찰
                  ↓
        전략 및 리스크 재검증
                  ↓
       실행 계획·자동화 정책(automation_level) 평가
                  ↓
   기존 시장별 승인 대기열 → 기존 시장별 주문 경로
```

AI스톡은 자동화 정책이 허용한 범위에서만 주문을 실행한다. 기본 정책은 분석·계획까지만 자동화하며, 자동 승인·주문은 정책·리스크·안전 조건을 통과한 전략에 한해 기존 시장별 경로로 수행한다(신규 브로커 우회 경로 없음).

## 1. 범위와 안전 원칙

### 1.1 구현 범위

신규 메인 경로 `/ai-stock`과 다음 9개 세부 탭을 추가한다.

1. 시장 브리핑
2. 내러티브
3. AI 종목 발굴
4. 종목 분석
5. 관찰종목
6. 포트폴리오
7. 성과 검증
8. AI 전략 관리
9. 실행 계획

AI스톡은 AI가 **시황 해석 · 종목 선정 · 매매 타이밍**을 주도하는 자동매매 시스템이다(§4.5). 종목 선정은 **ETF를 포함하고 소형주를 제외한 유니버스**(§4.6)에서, AI가 정리한 **월/주/일 시황 파일**(§4.7)을 컨텍스트로, **1차 배치 점수화 + 2차 실시간 타이밍**의 2단계(§4.8)로 처리한다. 자금·수량·실거래 허용은 항상 결정적 안전 계층(RiskEngine·정책·`automation_level`)이 게이팅한다.

세부 탭은 **9개로 고정**한다. 스케줄 자동 실행의 운영 가시성(스캔·스케줄 이력/실패)은 별도 탭을 늘리지 않고 탭3·탭8에 서브패널로 흡수한다(§5.3, §5.8, §5.12).

시장 필터는 `ALL`, `KR`, `US`를 지원한다. 단, 통화·거래시간·유동성·비용·DB 원천은 시장별로 분리한다.

### 1.2 안전 원칙

다음 기본값을 변경하지 않는다.

```text
DRY_RUN=true
TRADING_ENV=demo
ENABLE_LIVE_TRADING=false
REQUIRE_APPROVAL=true
```

AI스톡 후보의 실행 연결은 다음 순서만 허용한다.

```text
후보 발굴
→ 관찰종목 등록
→ 결정적 confirmed 검증
→ 승인된 실행 전략 연결
→ RiskEngine 검사
→ 실행 계획 생성
→ 기존 승인 대기열
→ 기존 OrderRouter
```

AI스톡은 처음부터 **완전 자동화 가능한 파이프라인**으로 설계한다. 다만 기본 운영값은 분석·계획까지만 허용하고, 전략별 자동화 정책이 명시적으로 허용한 경우에만 승인 큐 등록 또는 주문 실행까지 진행한다.

자동화 레벨:

```text
0 = 분석만 자동
1 = 후보 발굴 자동
2 = 관찰 등록 자동
3 = confirmed 자동 검증
4 = 실행 계획 자동 생성
5 = 승인 대기열 자동 등록
6 = 주문 자동 실행
```

기본값:

```text
automation_level=4
auto_approve=false
auto_execute=false
```

완전 자동 주문은 다음 조건을 모두 만족해야 한다.

- `automation_level >= 6`
- `auto_execute=true`
- 전략 상태가 `approved`
- `DRY_RUN=false`
- `ENABLE_LIVE_TRADING=true`
- `REQUIRE_APPROVAL=false` 또는 `auto_approve=true`
- 데이터가 fresh이고 stale/fallback 거래가 정책상 허용됨
- 점수·리스크·비중·손실 제한·시장시간 조건을 모두 통과

자동화 정책이 조건을 만족하지 않으면 해당 단계에서 멈추고 `blocked` 상태와 사유를 기록한다.

다음 상태에서는 실행 계획을 만들 수 없다.

- AI 평가 실패 후 안전한 fallback 결과가 없는 경우
- 가격·내러티브·시장 데이터가 stale인 경우
- 룰 최소 점수 미달
- 손절 기준 또는 무효화 조건 누락
- 리스크 검사 실패
- 승인된 실행 전략 미연결

### 1.2.1 운영 전환 단계 (demo → live)

AI스톡 자동화는 **데모 환경에서 먼저 운영**하고, 충분히 검증된 뒤에만 실제 계좌로 전환한다. 한 번에 실거래로 켜지 않는다.

| 단계 | 환경 설정 | 자동화 범위 | 목적 |
|---|---|---|---|
| 1. 데모 운영 | `TRADING_ENV=demo`, `ENABLE_LIVE_TRADING=false`, `DRY_RUN=true` | Level 0~4 (분석·후보·관찰·계획) | 점수·후보·자동화 흐름·안전 조건·무중단 검증 |
| 2. 데모 자동주문 검증 | `TRADING_ENV=demo`, 모의계좌 주문 허용 | Level 5/6 (demo 계좌) | 자동 승인·주문 경로, 리스크 한도, 일일 손실·중복·비중 한도 실측 |
| 3. 실계좌 전환 | `TRADING_ENV=live`, `ENABLE_LIVE_TRADING=true`, `DRY_RUN=false` | Level 6 (정책 허용 전략만) | 실거래 — **사용자 명시 승인 후에만** |

3단계(실계좌)로 전환하기 전 아래 게이트를 모두 충족해야 한다.

- 데모에서 일정 기간·건수의 자동 주문이 무사고로 동작
- 전략 상태 `approved`
- 일일 손실·종목/섹터/시장 비중·중복 주문 한도가 데모에서 실측 검증됨
- kill switch(`.runtime/kill_switch.json`)와 롤백 절차 확인
- AI한스톡(KR)·AI미스톡(US) 각 시장 계좌·API 키가 demo→live로 **올바르게 분리 전환**됨

주의:

- demo와 live의 계좌·API 키는 분리된 설정이며 혼용하지 않는다(VM `.env`와 로컬 `.env`도 별개).
- 실계좌 전환은 `TRADING_ENV`·`ENABLE_LIVE_TRADING`·`DRY_RUN`을 **함께** 변경해야 하며, 부분 변경 시 의도치 않은 실거래를 막기 위해 자동화 엔진이 시작 시 환경 정합성 검사를 한다(불일치 시 자동 주문 차단·`blocked` 기록).
- 현재 로컬 `.env`는 `TRADING_ENV=demo` + `ENABLE_LIVE_TRADING=false`이나 `DRY_RUN=false`이므로, 데모 자동주문 테스트 전에 `DRY_RUN` 의미와 모의계좌 동작을 먼저 확인한다.

### 1.3 AI한스톡과 AI미스톡 구분

AI스톡은 하나의 메인 메뉴이지만, 내부 데이터 계열은 **AI한스톡(국내)** 과 **AI미스톡(미국)** 으로 명확히 분리한다. 두 계열은 같은 화면·같은 9개 탭·같은 응답 schema를 공유하되, **데이터 원천·종목코드·통화·거래시간·지수·섹터·유동성·거래비용·세금·DB는 절대 혼합하지 않는다.**

| 구분 | **AI한스톡** | **AI미스톡** |
|------|-------------|-------------|
| 시장 코드 | `KR` | `US` |
| 대상 시장 | 한국(KOSPI/KOSDAQ) | 미국(NYSE/NASDAQ) |
| 데이터 공급자 | KIS API (`src/api/kis_api.py`) | 미스톡 소스 (`src/mistock/*`) |
| 종목 코드 | 6자리 숫자(`005930`) | 영문 티커(`AAPL`) |
| 통화 | KRW | USD |
| 거래시간 | 09:00–15:30 KST | 미 정규장(한국시간 환산 표기) |
| 시장지수 | KOSPI, KOSDAQ | S&P 500, Nasdaq, Russell 2000 |
| 섹터 분류 | 국내 섹터/테마 | GICS 섹터 |
| 유동성 기준 | 거래대금(원) | 거래대금(달러)/평균거래량 |
| 거래비용·세금 | 수수료 + 증권거래세(매도) | 수수료(+규제수수료), 거래세 없음 |
| 데이터 저장 | 한스톡 메인 DB (`src/db/repository.py`) | 미스톡 DB (`src/mistock/db.py`, `.runtime/mistock/trades.sqlite`) |
| 관심종목·유니버스 | 한스톡 관심종목, KIS 거래량 상위/조건검색 | 미스톡 관심종목, `NASDAQ_UNIVERSE`, 미스톡 스캔 후보 |
| 내러티브 | KR 내러티브(현재 기본 제공) | US 내러티브(소스 준비 후 활성화, 허위 생성 금지) |

분리 규칙:

- 후보·스캔·관찰·성과·실행계획의 모든 저장 행은 `KR`(AI한스톡) 또는 `US`(AI미스톡) 중 하나로 명시한다. `ALL`은 조회 집계용일 뿐 저장하지 않는다.
- AI한스톡 데이터와 AI미스톡 데이터를 직접 JOIN하거나 합산하지 않는다. 합산이 필요한 포트폴리오 화면에서만 `src/utils/exchange_rate.py`로 명시 환산하고 적용 환율·기준시각을 함께 표기한다.
- 시장 어댑터(`market_kr.py` = AI한스톡, `market_us.py` = AI미스톡)가 각 DB·공급자를 읽어 공통 도메인 객체로 변환하는 지점에서만 두 계열이 같은 형식으로 만난다.
- AI 전략·profile·feature/prompt 버전도 AI한스톡과 AI미스톡을 분리해 관리한다(시장에 맞지 않는 전략 선택 불가).
- 화면 상단 시장 필터는 `전체 / 한국(AI한스톡) / 미국(AI미스톡)`으로 라벨링한다.

### 1.4 설정 가능성 원칙 (모든 기준은 환경설정에서 변경)

**하드코딩 금지.** 점수 가중치·임계값·유니버스 기준(시총/거래대금/가격)·리스크 한도·TTL·스캔 주기·`automation_level`·데이터 신선도 기준 등 **모든 운영 수치와 처리 기준은 코드에 박지 않고 설정 계층에서 로드**한다. 본 계획서에 표기된 모든 수치(§4.4 가중치, §4.6 유니버스, §1.2 레벨, §9 TTL 등)는 **기본 제안값**이며 운영 중 변경 가능하다.

설정 계층(아래로 갈수록 우선 적용):

| 계층 | 저장 위치 | 담는 값 | 변경 방법 |
|---|---|---|---|
| 1 시스템 환경변수 | `.env` / `src/config.py` | 안전 가드(`DRY_RUN`·`ENABLE_LIVE_TRADING`·`REQUIRE_APPROVAL`·`TRADING_ENV`), API 키, 전역 기본 TTL·주기 | 환경설정 화면 `/env-settings` 또는 `.env` |
| 2 전략 profile | `ai_strategies.profile_json` | 점수 가중치, 최소 룰점수·confidence, feature/prompt version | 탭8 전략 편집 |
| 3 자동화·유니버스 정책 | `ai_stock_automation_policies`(§6.6) | `automation_level`, 리스크 한도, 유니버스 기준(시총·거래대금·가격), ETF/소형주 규칙, 시황·타이밍 임계 | 탭8 정책 편집 |

기존 환경설정 인프라(`src/dashboard/routes/settings.py`·`src/dashboard/services/env_service.py`)가 `.env`를 안전하게 읽고 쓴다. **AI스톡 전용 설정 그룹을 이 화면에 추가**하고 새 설정 화면을 따로 만들지 않는다.

규칙:

- 상위 계층이 하위를 override한다(시장별 정책 > 전략 profile > 전역 기본값). 미설정 항목은 하위 기본값을 사용한다.
- 모든 설정 항목은 **기본값·허용범위·검증**을 가진다. 범위 밖 값은 거부하고 사유를 반환한다.
- 안전 가드(`DRY_RUN`·`ENABLE_LIVE_TRADING`·`REQUIRE_APPROVAL`·자동주문 허용)는 **환경변수 계층에서만** 변경하며, 전략·정책이 이를 우회할 수 없다.
- 설정 변경은 감사 로그를 남기고, 자동화·전략 정책 변경 시 관련 검증·승인 상태를 무효화한다(§5.8).
- 코드 리뷰에서 신규 매직 넘버(하드코딩 임계값)는 거부하고 설정 계층으로 이동시킨다.

## 4. 공통 데이터 계약

### 4.1 시장 코드

```text
ALL | KR | US
```

`KR`은 **AI한스톡**, `US`는 **AI미스톡** 계열을 가리킨다(§1.3). `ALL`은 조회용 집계 값이다. 개별 후보와 저장 행에는 반드시 `KR`(AI한스톡) 또는 `US`(AI미스톡)만 저장한다.

### 4.2 후보 판단

```text
strong_watch | watch | neutral | avoid | insufficient_data
```

AI스톡 후보 판단에는 `buy`와 `sell`을 사용하지 않는다.

### 4.4 점수 산식

모든 구성 점수는 0~100 범위로 정규화한다.

```text
base_score =
    rule_score       × rule_weight
  + technical_score  × technical_weight
  + momentum_score   × momentum_weight
  + narrative_score  × narrative_weight
  + ai_score         × ai_weight

final_score =
    clamp(
      base_score
      × regime_multiplier
      × data_quality_multiplier
      - risk_penalty,
      0,
      100
    )
```

기본 가중치:

| 구성 | 기본값 |
|---|---:|
| 룰 | 0.20 |
| 기술 | 0.20 |
| 모멘텀 | 0.15 |
| 내러티브 | 0.25 |
| AI | 0.20 |

제약:

- 가중치 합은 1.0이어야 한다.
- AI confidence 미달 또는 fallback이면 AI 가중치를 0으로 만들고 나머지 가중치를 재정규화한다.
- 룰 최소 점수 미달 후보는 `watch` 이상으로 승격할 수 없다.
- stale 또는 필수 데이터 누락은 `insufficient_data`로 처리한다.
- 위험 점수는 최종 점수를 높일 수 없다.
- 같은 입력은 같은 결정적 점수를 반환해야 한다.

## 4.5 AI 주도 의사결정 모델

AI는 보조 점수 한 항목(`ai_score`)에 머물지 않고, **시황 해석 → 종목 선정 → 매매 타이밍 → 강도 제안**의 4개 핵심 판단을 주도한다. 단, **자금·수량·실거래 허용은 항상 결정적 안전 계층**(RiskEngine·정책·`automation_level`)이 최종 결정한다. AI는 "무엇을·언제·얼마나"를 제안·주도하고, 안전 계층은 "실제로 집행 가능한지"를 게이팅한다.

| AI 주도 영역 | AI가 하는 일 | 결정적 안전 계층이 막는 것 |
|---|---|---|
| ① 시황 해석 | 월/주/일 시황을 종합해 국면·테마 로테이션·리스크 판단(§4.7) | 데이터 stale/부족 시 국면 임의 생성 금지 |
| ② 종목 선정 | 유니버스(§4.6)에서 후보 점수화·순위·근거 생성(1차, §4.8) | 룰 최소 점수 미달 승격 차단, 소형주 제외 |
| ③ 매매 타이밍 | 후보 풀에 대한 실시간 진입/청산 신호(2차, §4.8) | stop·무효화·유동성·중복·일일손실 게이트 |
| ④ 강도 제안 | 목표 비중·확신도 제안 | 수량은 RiskEngine, 한도는 정책(§6.6) |

원칙:

- AI 주도성을 높이되 **설명가능성**을 유지한다. 모든 AI 판단은 근거(positive/negative factors, 연결 내러티브·시황 파일 참조)를 함께 저장한다.
- AI 실패·저신뢰·차단 시 룰 기반으로 **자동 강등**한다(§5.11.2). AI가 죽어도 시스템은 룰로 계속 동작한다.
- AI는 미래 가격·수익률을 확정 제시하지 않는다. `confidence`는 신뢰 수준이며 상승확률이 아니다.

## 4.6 종목 유니버스 정책 (ETF 포함 · 소형주 제외)

종목 선정의 출발점인 유니버스를 시장별로 명확히 정의한다. **ETF를 포함하고 소형주를 제외**한다.

- **포함:** 개별주 + ETF(섹터·테마·지수·채권 ETF). ETF는 `instrument_type='etf'`로 태깅해 개별주와 리스크·유동성 기준을 분리한다.
- **제외:** 소형주(저시총·저유동성), 관리종목·거래정지·우선주·스팩, 페니주, 신규상장 직후(데이터 부족) 종목.

| 기준(전략 profile에서 조정) | AI한스톡(KR) 기본값 | AI미스톡(US) 기본값 |
|---|---|---|
| 최소 시가총액 | 3,000억 원 | 20억 달러 |
| 최소 일평균 거래대금 | 50억 원 | 2,000만 달러 |
| 최소 주가 | 2,000원 | 5달러 |
| ETF 최소 AUM/거래량 | 별도 ETF 기준 적용 | 별도 ETF 기준 적용 |
| 제외 | 관리·정지·우선주·스팩·신규상장 | 페니주·저유동성·신규상장 |

- 수치는 **기본 제안값**이며 전략 profile/자동화 정책에서 시장별로 조정한다.
- 유니버스 구성 결과(통과·제외 사유)를 스캔 이력에 남겨 추적한다.
- ETF와 개별주는 같은 점수 schema를 쓰되, ETF는 내러티브·종목 과열 항목 대신 섹터/지수 추종·괴리율 항목을 사용한다.

## 4.7 시황 브리핑 파일 파이프라인 (월/주/일)

AI가 정리한 시황을 **파일로 영속화**하고, 이를 종목 점수화·타이밍의 컨텍스트로 재사용한다.

- **저장 위치:** `.runtime/ai_stock/briefings/{market}/` 아래에 `monthly/YYYY-MM.{json,md}`, `weekly/YYYY-Www.{json,md}`, `daily/YYYY-MM-DD.{json,md}`. (런타임 산출물이므로 커밋하지 않음, `.gitignore`)
- **이중 포맷:** `json`(기계 입력: 국면·점수·테마·리스크·관전포인트 구조화) + `md`(사람 열람용 요약). 내러티브 collector의 `save_json_file` 패턴을 재사용한다.
- **계층적 컨텍스트:** 일간 브리핑은 주간을, 주간은 월간을 참조한다. 1차 종목 점수화·2차 타이밍은 해당 시장의 최신 일간 브리핑(+ 상위 주간/월간)을 입력으로 사용한다.
- **내용:** 시장 국면(regime), 섹터·테마 로테이션, 매크로/이벤트 리스크, 변동성·breadth, "다음 기간 관전 포인트", 데이터 기준시각·신선도.
- **시장 분리:** AI한스톡/AI미스톡 브리핑을 폴더로 분리하고 통화·지수·섹터를 혼합하지 않는다.
- **생성 주체:** 탭1 시장 브리핑(§5.1)과 스케줄 잡이 생성한다. 일간은 장 시작 전/마감 후, 주간·월간은 주기 경계에 생성한다.
- **stale 처리:** 브리핑 파일이 기준 신선도를 넘으면 후보 confirmed·자동 타이밍에 사용하지 않고 배너로 표시한다.

## 4.8 2단계 분석: 1차 배치 점수화 / 2차 실시간 타이밍

종목 선정과 매매 타이밍을 **분리된 2단계**로 처리한다. 1차는 "무엇을 살지"(선정), 2차는 "언제 살지/팔지"(타이밍)를 담당한다.

| 구분 | 1차 — 배치 종목 점수화 (선정) | 2차 — 실시간 분석 (타이밍·집행) |
|---|---|---|
| 주기 | 일/주 배치(장전·마감 후) | 장중 실시간/단주기 폴링 |
| 입력 | 유니버스(§4.6) + 시황 파일(§4.7) + 지표·내러티브 | 1차 후보 풀 + 실시간 시세·호가·거래량 |
| AI 역할 | 종목별 점수화·순위·선정 근거(무거운 AI 호출) | 후보 풀 한정 경량 재평가·진입/청산 신호 |
| 산출 | `ai_stock_candidates` → 관찰종목 후보 풀 | 진입/청산 타이밍 신호 → 실행 계획/자동매매 |
| 실시간 소스 | — | KR: `KISWebSocketClient`(실시간 시세·체결), US: 폴링 주기 대체 |
| 탭 매핑 | 탭1·2·3·4·5 | 탭9 + 자동화 엔진(§5.12) |

규칙:

- 2차는 **1차가 선정한 후보 풀에 대해서만** 동작한다(실시간 비용·노이즈 통제). 풀에 없는 종목을 실시간으로 즉흥 매수하지 않는다.
- 2차 진입 신호: 돌파/되돌림/모멘텀 확인 등 결정적 트리거 + AI 타이밍 확신도. 청산 신호: `stop_price`·목표·무효화 조건·시간 청산.
- 2차 신호도 `automation_level`·리스크·정책 게이트를 거쳐야 주문으로 이어진다(§1.2, §5.12). 신호 자체가 곧 주문이 아니다.
- 실시간 데이터가 stale/단절이면 2차는 신규 진입을 멈추고 보유 보호(청산·stop)만 유지한다.
- 2차 타이밍 신호는 `ai_stock_timing_signals`(§6.8)에 기록해 1차 점수와 분리 추적한다.

시장별 실시간 차이(한계 명시):

- **AI한스톡(KR):** `KISWebSocketClient` 실시간 시세·체결로 촘촘한 2차 타이밍이 가능하다.
- **AI미스톡(US):** 현재 실시간 웹소켓이 검증되지 않아 **폴링 기반(`realtime_poll_seconds`)**으로 동작하며, KR 대비 지연이 크다. 미국 정규장은 한국 야간이므로 무인 운영 시 알림·자동화 정책을 별도로 두고, 1차 배치 선정 중심으로 운영하다 실시간 소스가 확보되면 2차를 강화한다.
- 시장별 2차 실시간 가능 범위는 설정(`realtime_poll_seconds` 등, §1.4)으로 조정하며, 미지원 구간은 2차를 비활성하고 1차 후보·계획까지만 제공한다.

## 5. 세부 탭별 구현 정의

## 5.1 탭 1 — 시장 브리핑

### 목적

한국·미국 시장 국면을 공통 형식으로 계산하고 후보 점수의 국면 승수에 사용한다.

### 데이터 원천

- KR (AI한스톡): KOSPI/KOSDAQ 지수 또는 현재 사용 가능한 국내 시장 데이터
- US (AI미스톡): 미스톡 가격 데이터와 사용 가능한 대표 지수 데이터
- 공통: 가격 이력, 이동평균, 변동성, breadth 가능 여부

외부 데이터가 필요한 지표는 선택 항목으로 둔다. 데이터가 없을 때 임의 중립값을 만들지 않는다.

### API

```text
GET  /api/ai-stock/overview?market=KR|US|ALL
GET  /api/ai-stock/briefings?market=&period=monthly|weekly|daily   # 시황 파일(§4.7)
POST /api/ai-stock/briefings/generate?market=&period=
```

### 주요 응답

- 시장 국면
- 추세 점수
- 변동성 점수
- breadth 점수 또는 `null`
- 긍정 요인
- 위험 요인
- 데이터 품질과 기준시각
- 최신 월/주/일 시황 파일 요약과 신선도(§4.7)

### 시황 파일 생성 (§4.7)

탭1은 AI 시황 브리핑(월/주/일)을 생성·열람하는 주체다. 일간은 장 시작 전/마감 후, 주간·월간은 주기 경계에 생성하며 `.runtime/ai_stock/briefings/{market}/`에 저장한다. 이 파일은 1차 종목 점수화·2차 타이밍의 컨텍스트 입력이 된다.

### 완료 조건

- KR과 US가 동일 schema를 반환한다.
- 필수 지표 누락 시 `insufficient_data`를 반환한다.
- 산출된 `regime_multiplier`가 후보 점수 계산에 실제 사용된다.
- 시장별 데이터가 서로 섞이지 않는다.
- 월/주/일 시황 파일이 생성되고, stale 시 1차·2차에 사용되지 않는다.

## 5.2 탭 2 — 내러티브

### 목적

기존 내러티브 데이터를 AI스톡에서 조회하고 종목 후보 점수에 연결한다.

### 데이터 원천

- `src/strategy/narrative_collector.py`
- `src/strategy/narrative_momentum.py`
- `config/theme_map.json`
- `.runtime/narrative_history.json`

초기 MVP에서는 기존 내러티브가 KR 중심인지 명시하고, US 데이터가 검증되지 않았으면 US 내러티브를 허위 생성하지 않는다. US 내러티브는 시장별 수집·매핑 소스가 준비된 후 활성화한다.

### API

```text
GET /api/ai-stock/narratives?market=KR|US|ALL
GET /api/ai-stock/narratives/{narrative_id}
```

### 점수 요소

- 강도
- 변화 방향
- 최신성
- 출처 수
- 감성
- 지속성·가속도
- 테마·섹터·종목 매칭 근거

### 제한

- `fading`, `disappeared`는 양수 보너스를 받지 않는다.
- 단일 출처는 감점한다.
- stale 데이터는 후보 확정에 사용할 수 없다.
- 문자열 유사도만으로 종목 연결을 확정하지 않는다.
- 연결 근거와 매칭 점수를 저장한다.

### 완료 조건

- 기존 내러티브 계산을 복제하지 않는다.
- 내러티브 데이터의 시장, 출처, 기준시각을 노출한다.
- 후보의 `narrative_score`에서 어떤 내러티브가 반영됐는지 추적 가능하다.

## 5.3 탭 3 — AI 종목 발굴

### 목적

시장 데이터, 룰 점수, 기술·모멘텀, 내러티브, AI 평가, 위험 감점을 결합해 관찰 후보 순위를 만든다.

### 유니버스

KR (AI한스톡):

- 한스톡 관심종목
- KIS 거래량 상위
- KIS 조건검색
- 전략별 유니버스
- 내러티브 테마 연결 종목

US (AI미스톡):

- 미스톡 관심종목
- `NASDAQ_UNIVERSE`
- 미스톡 스캔 후보
- 유동성 기준 통과 종목
- 향후 US 내러티브 연결 종목

위 유니버스에는 **유니버스 정책(§4.6)**을 적용한다: ETF를 포함(`instrument_type='etf'` 태깅)하고, 소형주(시총·거래대금·가격 하한 미달)·관리/정지/우선주/스팩/페니주·신규상장을 제외한다. 기준 수치는 정책(§6.6)에서 시장별로 변경 가능하며, 제외 종목과 사유를 스캔 이력에 남긴다. 이 단계가 **1차 배치 종목 선정**(§4.8)에 해당한다.

### 처리 순서

```text
유니버스 구성 (§4.6 정책: ETF 포함·소형주 제외)
→ 중복 제거·시장 코드 정규화
→ 시황 파일 컨텍스트 로드 (§4.7)
→ 가격/거래량 데이터 품질 검사
→ 룰 최소 조건 검사
→ 구성 점수 계산
→ 선택적 AI 평가
→ 위험 감점
→ 시장 국면 보정
→ 판단 상태 결정
→ 스캔·후보 저장 (1차 후보 풀)
```

### API

```text
POST /api/ai-stock/scans
GET  /api/ai-stock/scans/{scan_id}
GET  /api/ai-stock/candidates?market=&scan_id=&decision=&min_score=
```

### 스캔 상태

```text
queued | running | completed | partial | failed
```

동일 `(market, strategy_id)`의 실행 중 스캔은 중복 생성하지 않고 HTTP 409를 반환한다.

### 스캔 진행·이력 패널 (운영 가시성 흡수)

별도 운영 탭을 만들지 않고, 탭3 하단에 **스캔 진행·이력 서브패널**을 둔다(수동 실행과 스케줄 자동 실행 §5.12을 같은 곳에서 확인).

- 현재 스캔 진행 상태(queued/running/…)와 단계 텍스트(유니버스→점수→AI평가)
- 최근 스캔 이력: 시장(AI한스톡/AI미스톡)·전략 버전·시작/완료 시각·후보 수·fallback 수·상태·실패 사유(`error_message`)
- 중복 차단(409) 발생 표시와 마지막 스케줄 실행 시각
- 이력 데이터는 `ai_stock_scans` + `scheduler_results`에서 읽는다(신규 저장소 없이 재사용)

### 완료 조건

- 룰 기준 미달 후보는 AI 점수로 승격되지 않는다.
- AI 오류는 후보 전체 실패가 아니라 명시적 fallback으로 처리한다.
- 모든 점수는 0~100 범위다.
- 후보 결과에 시장·전략·모델·feature·prompt·profile 버전이 기록된다.
- 후보 발굴 단계는 주문·승인을 직접 만들지 않는다. 주문·승인은 실행 계획(탭9)과 자동화 정책 엔진(§5.12)을 거쳐, 정책이 허용한 경우에만 기존 시장별 경로로 수행한다.

## 5.4 탭 4 — 종목 분석

### 목적

후보 한 종목의 점수 구성과 판단 근거를 재현 가능하게 보여준다.

### API

```text
GET /api/ai-stock/candidates/{candidate_id}
```

### 화면 영역

- 현재 가격과 데이터 기준시각
- 5개 구성 점수와 위험 점수
- 기술 분석
- 가격·거래량 모멘텀
- 시장·섹터 상대강도
- 연결 내러티브와 매칭 근거
- 긍정·부정 요인
- 확인 조건과 무효화 조건
- fallback 여부와 이유
- 전략·모델·profile 버전

### 제한

- 목표가격과 기대수익률을 확정적으로 표현하지 않는다.
- 입력값 요약과 실제 모델 기여도를 구분한다.
- 손절 참고값은 전략 또는 ATR 기반 계산식과 기준시각을 함께 표시한다.

### 완료 조건

- 목록 점수와 상세 점수가 동일한 저장 결과를 사용한다.
- UI에서 점수를 재계산하지 않는다.
- 후보 선정·제외 이유를 사람이 검토할 수 있다.

## 5.5 탭 5 — 관찰종목

### 목적

AI 후보를 즉시 실행하지 않고 일정 기간 추적하며 결정적 조건 충족 여부를 검사한다.

### 상태

```text
discovered
→ watching
→ confirmed
→ execution_planned

discovered | watching
→ rejected | expired
```

### API

```text
GET    /api/ai-stock/watchlist
POST   /api/ai-stock/watchlist
PATCH  /api/ai-stock/watchlist/{candidate_id}
DELETE /api/ai-stock/watchlist/{candidate_id}
```

### confirmed 조건

- 최소 최종 점수
- 최소 룰 점수
- 최대 위험 점수
- fresh 데이터
- 유동성 기준
- 유효한 확인·무효화 조건
- 승인된 실행 전략 존재
- 진입 참고가와 손절 기준 존재

사용자가 UI에서 상태를 강제로 `confirmed`로 바꾸더라도 서버가 위 조건을 다시 검사한다.

### 완료 조건

- 허용되지 않은 상태 전이는 HTTP 409로 거부한다.
- AI 응답만으로 `confirmed`가 되지 않는다.
- stale 후보는 `confirmed`가 될 수 없다.
- 상태 변경 이력을 기록한다.

## 5.8 탭 8 — AI 전략 관리

### 목적

AI스톡 점수 profile, 모델, 버전, 검증 상태를 기존 AI 전략 생명주기와 일관되게 관리한다.

### 데이터 원천

- KR 전략 (AI한스톡): `src/db/strategy_repository.py`
- US 전략 (AI미스톡): 현재 미스톡 `ai_strategies` 구조

MVP에서는 AI스톡 전용 profile이 기존 전략을 참조하도록 하고, AI한스톡/AI미스톡 전략 저장소를 즉시 통합하지 않는다. 장기적으로 공통 저장소 통합 여부를 별도 결정한다. 전략 선택 시 시장이 일치해야 하며, AI한스톡 후보에 AI미스톡 전략을 적용할 수 없다.

### API

```text
GET  /api/ai-stock/strategies?market=KR|US
POST /api/ai-stock/strategies/{strategy_id}/validate
POST /api/ai-stock/strategies/{strategy_id}/select
```

전략 생성·수정·승인 전체 API를 중복 구현하지 않고 기존 시장별 API 또는 저장소 서비스를 호출한다.

### 스케줄 설정·실행 이력 패널 (운영 가시성 흡수)

별도 운영 탭 대신, 탭8에 **스케줄 설정·실행 이력 서브패널**을 둔다(전략 단위로 스케줄을 관리하는 기존 내러티브 패턴과 동일).

- 시장별 스케줄 설정: AI한스톡/AI미스톡 각 `strategy_id`의 enabled, 실행 윈도우(KR 마감 후 / 미국 장중), 주기(interval) — 기존 `strategy_schedules` + `save_strategy_schedule()`/`load_strategy_schedule()` 재사용
- 디스패치 실행 이력: `dispatch_due_schedules()` 실행 결과, 마지막 실행 시각(`mark_strategy_schedule_run`), 성공/실패·fallback 여부
- 안전 표시: 현재 전략의 `automation_level`, `auto_approve`, `auto_execute`와 다음 차단 단계를 패널에 명시한다. 기본 정책에서는 주문·승인을 자동 생성하지 않지만, Level 5/6 전략은 정책 범위 내 자동 승인·주문이 가능함을 구분 표기한다(§5.12.4).

### 검증 항목

- 가중치 합
- 점수·입출력 schema
- 모델 provider 설정
- timeout·fallback
- feature/prompt/profile version
- rule-only 비교
- out-of-sample 결과
- profile hash 일치

### 완료 조건

- 전략 변경 시 검증·승인 결과가 무효화된다.
- 시장에 맞지 않는 전략을 선택할 수 없다.
- AI스톡 후보에 실제 사용된 전략 버전과 hash가 저장된다.

## 5.9 탭 9 — 실행 계획

### 목적

검증된 관찰 후보를 기존 실행 계획 및 승인 대기열에 안전하게 연결하고, **2차 실시간 타이밍 신호(§4.8)를 이 탭에서 모니터링**한다. 탭9는 1차 후보 풀이 실제 집행으로 이어지는 종착 화면이다.

### API

```text
GET  /api/ai-stock/execution-plans?market=KR|US
POST /api/ai-stock/execution-plans
POST /api/ai-stock/execution-plans/{plan_id}/queue-approval
GET  /api/ai-stock/timing-signals?market=&candidate_id=   # 2차 실시간 신호(§6.8)
```

### 2차 실시간 타이밍 신호 패널 (§4.8 연결)

탭9 상단에 2차 실시간 신호를 표시해, "선정된 후보가 지금 진입/청산 타이밍인가"를 한 화면에서 본다(F40).

- 신호 목록: 종목·`signal_type`(entry/exit/hold)·`trigger`·`ai_timing_confidence`·`decision`(proceed/blocked)·`blocked_reason`·기준시각
- 신호는 **1차 후보 풀 종목만** 표시(즉흥 신호 없음, §4.8)
- 각 신호가 정책·리스크 게이트를 통과했는지(`proceed`/`blocked`)와 다음 단계(계획 생성/승인/주문)를 함께 표시
- 실시간 단절·stale 시 배너로 "신규 진입 중단, 청산만" 상태를 명시(§4.8)
- KR은 실시간, US는 폴링 지연을 라벨로 구분 표기

계획 생성, 승인 대기열 전송, 주문 실행은 단계적으로 분리한다. 기본값에서는 계획 생성만으로 주문이나 승인이 발생하지 않는다. 단, 전략별 자동화 정책(§1.2, §6.6)이 `automation_level >= 5`이면 승인 대기열 등록을 자동화할 수 있고, `automation_level >= 6`이면 기존 시장별 주문 경로를 통해 주문 실행까지 자동화할 수 있다.

### 승인 대기열 연결 경계

AI스톡 실행 계획은 메인 DB의 `ai_stock_execution_plans`에 먼저 저장한다. 승인 대기열 전송은 사용자의 별도 동작이며, 시장별 기존 승인 경로를 절대 혼합하지 않는다.

| 시장 | 계획 저장 | 승인 전송 대상 | 주문 경로 |
|---|---|---|---|
| `KR` AI한스톡 | 메인 DB `ai_stock_execution_plans` | 메인 DB `approvals` / 기존 한스톡 승인 서비스 | 기존 한스톡 `OrderRouter` |
| `US` AI미스톡 | 메인 DB `ai_stock_execution_plans` | 미스톡 DB `approvals` / `src/dashboard/routes/mistock.py` 승인 흐름 | 기존 미스톡 주문 경로 |

규칙:

- `US` 계획을 메인 DB `approvals`에 직접 넣지 않는다.
- `KR` 계획을 미스톡 DB `approvals`에 넣지 않는다.
- `ai_stock_execution_plans`에는 `approval_market`, `approval_db`, `approval_id`, `approval_status`를 저장해 어느 승인 큐로 전송됐는지 추적한다.
- 승인 전송 실패는 계획 상태를 `approval_queue_failed`로 남기며 주문 시도로 해석하지 않는다.
- 1차 구현에서 US 승인 연결이 충분히 검증되지 않으면, `US`는 계획 생성까지만 허용하고 `queue-approval`은 HTTP 409와 사유를 반환한다.
- 자동 승인/주문도 반드시 위 시장별 큐와 주문 경로를 사용한다. AI스톡 서비스가 브로커 API를 직접 호출하는 신규 우회 경로를 만들지 않는다.
- 자동 주문 실행 결과는 `ai_stock_execution_runs`에 별도 기록하고, 기존 주문·체결·승인 감사 로그와 연결한다.

### 생성 조건

- 관찰 상태 `confirmed`
- 승인된 실행 전략
- fresh 데이터
- 진입 참고가와 유효한 `stop_price`
- `entry_price > stop_price`
- 리스크 예산과 현금 한도
- 종목·섹터·전략·시장 비중 한도
- 일일 손실 제한
- 중복 계획·주문 없음
- 시장 거래 가능 여부

### 수량 계산

```text
risk_budget = operating_capital × max_risk_per_trade_pct
risk_per_share = entry_price - stop_price
quantity = floor(min(risk_budget / risk_per_share, cash_budget / entry_price))
```

거래비용과 최소 주문금액을 적용한 후 수량이 0이면 계획을 만들지 않는다.

### 완료 조건

- AI스톡 서비스에서 브로커 API를 직접 호출하지 않는다. `KR` 자동 실행은 기존 한스톡 주문 경로, `US` 자동 실행은 기존 미스톡 주문 경로로만 수행한다.
- 안전 검사별 통과·실패 코드와 사유를 저장한다.
- 기본값에서는 계획 생성 후 별도 사용자 동작으로 승인 대기열에 보낸다. 자동화 정책이 허용한 전략만 승인 대기열 등록 또는 주문 실행을 자동 수행한다.
- 기존 승인·주문 감사 로그를 그대로 사용한다.

## 5.10 서브탭 간 데이터 흐름과 유기적 연결

9개 탭은 독립 화면이 아니라 **하나의 분석 파이프라인**으로 연결된다. 각 탭의 출력은 다음 탭의 입력이 되며, 공통 연결 키(`scan_id`, `candidate_id`, `regime_multiplier`, `narrative_id`, `strategy_id`/`version`, `profile_hash`)로 추적 가능하게 묶인다. **이 연결은 AI한스톡(KR)과 AI미스톡(US) 계열별로 독립 실행되며, 두 체인은 시장 코드로 분리된 채 흐른다.** (집계 조회 `ALL`에서만 나란히 표시)

### 5.10.1 파이프라인

```text
[탭8 AI 전략 관리] ──(profile: 가중치·model·feature/prompt version·profile_hash)──┐
                                                                                 │ 모든 탭이 동일 profile 버전 참조
[탭1 시장 브리핑] ──(regime, regime_multiplier, data_quality)──┐                 │
[탭2 내러티브]    ──(narrative_id, narrative_score, 매칭근거)──┤                 │
                                                              ▼                 ▼
                                              [탭3 AI 종목 발굴: 점수 결합·순위]
                                                              │  scan_id, candidate_id 생성
                                                              ├──▶ [탭4 종목 분석] (candidate_id로 동일 저장결과 상세 표시)
                                                              └──▶ [탭5 관찰종목] (후보 등록 → discovered)
                                                                          │ confirmed (결정적 검증)
                                                                          ├──▶ [탭6 포트폴리오] (관찰후보 + 보유 → 목표 비중·집중도)
                                                                          │            │ 비중 한도
                                                                          └──▶ [탭9 실행 계획] ◀───────────┘
                                                                                       │ confirmed + 비중/리스크 통과
                                                                                       ▼
                                                                            기존 승인 대기열 → OrderRouter
[탭3~탭5 후보] ──(생성 후 경과 거래일 가격)──▶ [탭7 성과 검증] ──(rule-only 대비 악화 시)──▶ [탭8 review_required 제안]
```

### 5.10.2 탭별 입력·출력·연결 키

| 탭 | 상류 입력(어디서) | 핵심 처리 | 하류 출력(어디로) | 연결 키 |
|----|------------------|-----------|------------------|---------|
| 1 시장 브리핑 | 시장지수·가격(KIS/미스톡) | 시장 국면 산출 | `regime_multiplier`→탭3 점수 보정 | `market`, `data_as_of` |
| 2 내러티브 | narrative_collector/momentum | 내러티브 점수·종목 매칭 | `narrative_score`·`narrative_id`→탭3·탭4 | `market`, `narrative_id` |
| 3 AI 종목 발굴 | 탭1 국면 + 탭2 내러티브 + 지표/모멘텀 + 탭8 profile | 5개 점수 결합→최종 순위 | 후보 생성→탭4·탭5 | `scan_id`, `candidate_id`, `profile_hash` |
| 4 종목 분석 | 탭3 후보(`candidate_id`) | 저장 점수 재현·근거 표시(재계산 금지) | 관찰 등록 트리거→탭5 | `candidate_id` |
| 5 관찰종목 | 탭3/탭4 후보 | 결정적 confirmed 검증·상태 추적 | confirmed→탭6·탭9 | `candidate_id`, `status` |
| 6 포트폴리오 | 탭5 관찰후보 + 보유 잔고 | 목표 비중·집중도·위험예산 | 비중 한도→탭9 | `market`, `candidate_id` |
| 7 성과 검증 | 탭3~5 후보의 사후 가격 | AI vs rule-only 성과 측정 | 악화 시 review 제안→탭8 | `candidate_id`, `strategy_id`/`version` |
| 8 AI 전략 관리 | 운영자 입력 + 탭7 성과 | profile·가중치·버전·검증 | profile→탭1~탭9 전체 | `strategy_id`, `version`, `profile_hash` |
| 9 실행 계획 | 탭5 confirmed + 탭6 비중 + 탭8 전략 | RiskEngine 수량·안전검사 | 기존 승인 대기열 | `candidate_id`, `plan_id`, approval ID |

### 5.10.3 연결 무결성 규칙

- 모든 후보·관찰·성과·계획 행은 자신을 만든 `scan_id`·`profile_hash`·`strategy_version`을 보유해, 어떤 국면·내러티브·전략 버전으로 산출됐는지 **역추적**할 수 있다.
- 상류 데이터가 stale이거나 누락이면 그 연결은 끊기고 하류 탭은 `insufficient_data`/배너로 표시하며, **confirmed·실행 계획으로 진행되지 않는다.**
- 탭4·탭6·탭7은 점수를 재계산하지 않고 탭3이 저장한 결과를 **단일 출처로 재사용**한다(화면 간 숫자 불일치 금지).
- AI한스톡 체인의 키와 AI미스톡 체인의 키는 같은 `candidate_id` 공간을 공유하되 `market`으로 항상 구분되며, 두 시장의 후보가 같은 실행 계획·포트폴리오 비중 한도에 직접 혼합되지 않는다.

## 5.11 실제 AI·전략 사용 연결 규약 (코드 실측 기반)

계획상의 "AI 평가"·"전략"은 가상의 신규 모듈이 아니라 **이미 존재하는 코드**에 연결한다. 아래는 실제 구현(`src/strategy/predict.py`, `src/db/strategy_repository.py`, `src/config.py`)을 확인해 정리한 사실과 AI스톡 연결 규약이다.

### 5.11.1 실제 AI 평가기

- 모듈: `src/strategy/predict.py`의 `ModelPredictor`.
- 공급자·모델: `provider="openai_responses"`, 모델 `config.openai_model`(기본 `gpt-5-mini`), 엔드포인트 `https://api.openai.com/v1/responses`, `config.openai_timeout_seconds`(기본 20초).
- 설정값(`src/config.py`): `ai_strategy_enabled`(기본 `false`), `ai_score_weight`(기본 0.40), `ai_min_model_confidence`(기본 0.60), `openai_api_key`, `online_access_blocked`.
- 캐시: `.runtime/openai_ai_cache.json` (provider·model·feature 기반 키). AI스톡 캐시 키(§9)는 이를 감싸 시장·전략·profile을 추가한다.
- 상태값: `ready | low_confidence | disabled | fallback`.
  - `disabled`: `ai_strategy_enabled=false`
  - `fallback`: `openai_api_key` 미설정, `online_access_blocked`, 호출 예외
  - `low_confidence`: `probability < ai_min_model_confidence`

### 5.11.2 probability → ai_score / confidence 매핑 (안전 핵심)

`ModelPredictor.predict()`의 원천 출력은 `ml_score = probability`(상승확률 0~1)다. AI스톡은 **상승확률을 confidence로 노출하지 않는다**(§4.3, §8.5). 다음 규약으로 변환한다.

- AI스톡 `ai_score`(0~100)는 `probability`를 **점수화 입력**으로만 사용해 산출한다(예: 정규화·전략 profile 반영). 상승확률 그 자체를 화면 점수로 노출하지 않는다.
- AI스톡 `confidence`는 **모델 신뢰 수준**(model_status, 캐시 적중, 입력 데이터 품질)에서 별도 산출하며 `probability`와 동일시하지 않는다. UI 라벨에 "신뢰도(상승확률 아님)"를 병기한다.
- model_status별 점수 반영:
  - `ready` → `ai_score`를 `ai_score_weight`로 반영
  - `low_confidence` → **AI 가중치 0**, 룰 점수로 fallback (저신뢰 AI가 후보를 승격/강등 못 함 — 기존 코드 동작과 일치)
  - `disabled` → `ai_score` 제외, base_score를 룰/기술/모멘텀/내러티브로 재정규화
  - `fallback` → `fallback_used=true`, 룰 기반 점수로 후보 유지(스캔 중단 아님)
- 입력 features: AI스톡 feature 빌더가 시장 국면(탭1)·내러티브(탭2)·기술·모멘텀을 `features` dict로 만들어 `predict(features)`에 주입한다. 미국(AI미스톡)도 `src/strategy/indicators.py`(RSI/MACD/SMA/볼린저/ATR)를 그대로 공유하므로 동일 형식 features 생성이 가능하다.

### 5.11.3 실제 전략 lifecycle

- 저장소: `src/db/strategy_repository.py`의 `ai_strategies` 테이블·이벤트·성과 함수.
- AI스톡 상태 머신: `draft → static_verified → verified → backtested → approved → review_required | retired`.
- profile_hash·strategy_version: 기존 `strategy_profile_hash()`/`normalize_ai_strategy()`를 사용해 후보·스캔 행에 기록.
- AI한스톡/AI미스톡 전략은 `market`으로 분리해 선택하며, 시장 불일치 전략은 적용할 수 없다(§5.8).

## 5.12 스케줄 자동 실행과 무중단 운영 (코드 실측 기반)

"스케줄로 실행되면 알아서 오류 없이 작동"하려면, 기존 스케줄러에 **AI스톡 자동화 엔진을 안전하게 통합**해야 한다. 자동화 엔진은 전략별 `automation_level`에 따라 분석만 수행할 수도 있고, 실행 계획·승인 큐·주문 실행까지 진행할 수도 있다. 기존 구현을 확인한 결과와 통합 규약은 다음과 같다. (운영 가시성은 별도 탭을 만들지 않고 **탭3 스캔 진행·이력 패널**과 **탭8 스케줄 설정·실행 이력 패널**에 흡수한다 — 세부 탭은 9개 유지.)

### 5.12.0 현재 상태와 자동화 전제조건 (오해 방지)

> **이 절은 "목표 동작(계획)"이며, 지금 자동으로 도는 것이 아니다.** 실측 결과 `src/ai_stock`·AI스톡 라우터·`dispatch_due_schedules()`의 AI스톡 분기는 **모두 미구현(코드 0)**이다. 또한 로컬 `.env`는 `AI_STRATEGY_ENABLED=false`여서 AI 평가는 호출되지 않고 룰 점수만 사용된다.

자동 실행이 실제로 성립하려면 아래 6가지가 **모두** 충족돼야 한다. 하나라도 빠지면 "자동"이 아니라 수동 버튼 실행 또는 룰 fallback으로만 동작한다.

| # | 전제조건 | 현재 |
|---|---------|------|
| 1 | AI스톡 코드 구현(`src/ai_stock/*`, 라우터, DB 4테이블) | ❌ 미구현 |
| 2 | `dispatch_due_schedules()`에 AI스톡 분기 추가 | ❌ 없음 |
| 3 | `strategy_schedules`에 `ai_stock_kr/us`를 enabled로 등록 | ❌ 없음 |
| 4 | 주기 호출 주체 — VM은 cron(`scripts/vm/strategy-dispatch.sh` + `install-strategy-dispatch-cron.sh`), 로컬(Windows)은 작업 스케줄러 별도 필요 | ❌ 로컬 없음 |
| 5 | `AI_STRATEGY_ENABLED=true` + `OPENAI_API_KEY` + 온라인 접근 허용 | ⚠️ 키는 있으나 `enabled=false` |
| 6 | 안전값 점검(`DRY_RUN`·`REQUIRE_APPROVAL`·`TRADING_ENV`) | ⚠️ 점검 필요(로컬 `DRY_RUN=false`) |

실행 주체 요약:

- **로컬(Windows):** cron이 없고 대시보드에도 전략 디스패치 스레드가 없다(snapshot·auto-approval sweeper만 존재). 따라서 AI스톡 스캔은 **화면 버튼으로 수동 실행**한다.
- **VM(Linux):** cron이 `python -m src.strategy_scheduler`를 주기 호출할 때만 자동 실행된다.

### 5.12.1 기존 스케줄 인프라(실측)

- 디스패처: `src/strategy_scheduler.py`의 `dispatch_due_schedules()` — `strategy_schedules`(enabled) 테이블을 읽어 실행 윈도우·주기를 만족하는 전략만 실행하고, **각 전략을 try/except로 격리**하며 `mark_strategy_schedule_run()`으로 중복 실행을 막는다. 내러티브는 이미 별도 분기(`run_narrative_momentum_cycle`)로 처리한다.
- 실행 사이클: `src/scheduler.py`의 `run_scheduled_cycle()`은 내부에서 **`trader.run()`(주문 생성 경로)** 을 호출한다. 따라서 **AI스톡 스캔은 이 함수를 호출하면 안 된다.**
- 격리 집합: `_ISOLATED_STRATEGY_IDS`(현재 `plunge_bounce`, `heikin_ashi_scalping`)로 전략별 카테고리를 제한하는 패턴이 이미 있다.
- 미국 자동화: `src/mistock/scheduler.py`의 `run_mistock_scheduled_cycle()`·`is_us_market_open()`로 미국 장중 판정이 가능하다.

### 5.12.2 AI스톡 통합 규약

- AI한스톡·AI미스톡을 각각 별도 `strategy_id`(예: `ai_stock_kr_v1`, `ai_stock_us_v1`)로 `strategy_schedules`에 등록해 **시장별 독립 스케줄·독립 윈도우**로 돌린다.
- `dispatch_due_schedules()`에 AI스톡 분기를 추가하되, `run_scheduled_cycle`(기존 한스톡 주문 생성 경로)을 직접 호출하지 않고 **AI스톡 자동화 엔진**(`automation_service.run_strategy(market=..., strategy_id=...)`)을 호출한다. 자동화 엔진 내부에서 `automation_level`을 평가해 필요한 단계까지만 진행한다.
- AI스톡 `strategy_id`를 격리 집합에 포함해 `force_strategy_id`(선택 전략)와 충돌하지 않게 한다. 기본 Level 0~4에서는 주문 카테고리를 만들지 않고, Level 5/6에서만 시장별 기존 승인·주문 경로를 명시적으로 호출한다.
- 실행 윈도우: AI한스톡은 KR 장중/마감 후, AI미스톡은 `is_us_market_open()` 기반 미국 장중으로 분리한다.

### 5.12.3 무중단·오류 격리 (스케줄 안전)

- **시장 격리:** AI한스톡 스캔 실패가 AI미스톡 스캔에 전파되지 않도록 시장별 try/except로 분리한다(디스패처의 전략별 격리와 동일 원칙).
- **종목 부분 실패:** 스캔 내부에서 특정 종목의 데이터 부족·AI 오류는 후보 전체 실패가 아니라 `insufficient_data`/`fallback`으로 격리하고 스캔 상태를 `partial`로 기록한다(§5.3).
- **AI/네트워크 차단:** `online_access_blocked`이거나 `openai_api_key` 미설정이면 예외가 아니라 **룰 기반 fallback으로 스캔을 완료**한다(`fallback_used=true`). 스케줄이 멈추지 않는다.
- **중복 실행 방지:** `ai_stock_scans.status IN ('queued','running')` 저장소 락(동일 `(market, strategy_id)` 409, §5.3, §6.1) + `mark_strategy_schedule_run()` 주기 가드를 함께 사용한다.
- **재시도·기록:** 일시 오류는 기존 재시도 환경변수 패턴(`HANSTOCK_*_RETRIES`)을 따르고, 결과는 `scheduler_results` 테이블과 `.runtime/ai_stock/` 결과 파일에 기록한다(내러티브 패턴과 동일). 실패 사유를 `error_message`에 저장한다.

### 5.12.4 자동화가 절대 넘지 않는 경계

- 스케줄 자동 실행은 전략별 `automation_level`이 허용한 단계까지만 수행한다.
- 기본 정책(`automation_level=4`, `auto_approve=false`, `auto_execute=false`)에서는 **시장 브리핑·내러티브·후보 발굴·관찰 등록·confirmed 검증·실행 계획 생성**까지만 자동 수행하고 승인·주문은 자동 수행하지 않는다.
- `automation_level >= 5`이면 자동 승인 대기열 등록을 허용할 수 있다. 이때도 시장별 기존 승인 큐를 사용해야 하며, `KR`/`US` 큐를 혼합하지 않는다(§5.9).
- `automation_level >= 6`이면 자동 주문 실행을 허용할 수 있다. 이때도 실행 환경별 주문 허용 조건, 전략 `approved`, 리스크 정책 통과가 모두 필요하다(§1.2).
- stale·fallback·룰 미달 상태에서는 정책이 명시적으로 허용하지 않는 한 confirmed·실행 계획·승인·주문으로 진행하지 않는다. 기본값은 `allow_fallback_trade=false`, `allow_stale_data_trade=false`다.

### 5.12.5 성과 갱신 잡 (별도 주기)

- `ai_stock_performance`의 1·5·20거래일 수익률·MFE·MAE·벤치마크는 **스캔과 분리된 일일 잡**으로 갱신한다(후보 생성 시점에는 미래 가격이 없으므로).
- 미완료 표본은 평균에 포함하지 않으며(§5.7), 거래일 경과 후에만 채운다. 이 잡 역시 시장별로 분리해 AI한스톡/AI미스톡 가격·통화를 혼합하지 않는다.

### 5.12.6 탭별 자동 실행 선행조건·부족 시 처리

| 탭 | 자동 실행 시 선행 데이터 | 부족·실패 시 처리 |
|----|------------------------|------------------|
| 1 시장 브리핑 | 지수·가격 이력 | 필수 지표 누락 → `insufficient_data`, 후보 확정·실행 차단. 점수 계산은 참고용으로만 허용하고 `regime_multiplier`를 임의 중립값으로 생성하지 않음 |
| 2 내러티브 | 내러티브 history·theme_map | stale/US 미지원 → 감점 또는 비활성(허위 생성 금지) |
| 3 AI 종목 발굴 | 탭1·탭2 + 지표 + 전략 profile + AI키 | AI 차단 → 룰 fallback, 종목 부분 실패 → partial |
| 4 종목 분석 | 탭3 저장 후보 | 후보 없음 → 빈 상태 안내(재계산 안 함) |
| 5 관찰종목 | 탭3 후보 + 결정적 검증 입력 | 검증 입력 부족 → confirmed 차단, watching 유지 |
| 6 포트폴리오 | 보유 잔고 + 환율 | 잔고·환율 stale → 해당 자산 제외·사유 표기 |
| 7 성과 검증 | 후보 사후 가격(경과 거래일) | 표본 부족 → `insufficient_sample` |
| 8 AI 전략 관리 | `ai_strategies` profile | 검증 미충족 → 상태 승격 차단 |
| 9 실행 계획 | confirmed + 승인 전략 + stop_price + 자동화 정책 | 조건 미충족 → 계획/승인/주문 단계별 차단, `blocked_reason` 기록 |

## 6. 저장 구조

AI스톡 이력은 한스톡 메인 DB에 저장하는 것을 기본으로 한다. US 원천 데이터는 미스톡 DB에서 읽되, AI스톡 스캔 결과는 공통 비교를 위해 메인 DB에 시장 코드와 함께 저장한다.

### 6.0 migration 위치와 원칙

- 테이블 생성은 기존 초기화 흐름과 맞추기 위해 `src/db/repository.py:init_db()`에서 호출되도록 한다.
- 구현은 `src/db/ai_stock_repository.py`에 저장소 함수를 두되, `init_ai_stock_tables(conn)` 같은 helper를 제공하고 `repository.init_db()`가 이를 호출한다.
- 기존 운영 DB를 보존해야 하므로 destructive migration은 금지한다. 컬럼 추가는 기존 `_ensure_column()` 패턴을 따른다.
- SQLite 기준 FK 제약이 항상 강제된다고 가정하지 않는다. 서비스 계층에서 `scan_id`, `candidate_id`, `watchlist_id` 존재 여부를 재검증한다.
- 미스톡 DB에는 AI스톡 산출물을 중복 저장하지 않는다. 단, US 승인 전송 시에는 미스톡 기존 `approvals`에만 승인 요청 행을 만든다(§5.9).
- 모든 `market` 저장값은 `KR` 또는 `US`만 허용하고, DB 저장 함수에서 `ALL` 입력을 거부한다.

### 6.1 `ai_stock_scans`

- `id` (PK, 후보·응답에서 `scan_id`로 참조)
- `market`
- `strategy_id`
- `strategy_version`
- `model`
- `feature_version`
- `prompt_version`
- `profile_hash`
- `status`
- `started_at`
- `completed_at`
- `data_as_of`
- `candidate_count`
- `fallback_count`
- `error_message`

인덱스:

- `(market, started_at)`
- `(market, strategy_id, status)`

중복 실행 방지:

- 동일 `(market, strategy_id)`에 `queued` 또는 `running` 스캔이 있으면 새 스캔 생성을 거부한다.
- 스캔 생성은 단일 트랜잭션에서 "기존 running/queued 확인 → 새 row 생성" 순서로 처리한다.
- SQLite 동시 요청 race를 줄이기 위해 저장소 함수는 `BEGIN IMMEDIATE` 또는 동등한 쓰기 락을 사용한다.
- 프로세스가 죽어 `running`이 남을 수 있으므로 `started_at` 기준 stale-running TTL을 두고, TTL 초과 시 `failed` 또는 `abandoned`로 정리한 뒤 새 스캔을 허용한다.

### 6.2 `ai_stock_candidates`

- 식별자와 스캔 FK
- 시장·종목·통화·가격
- 구성 점수와 최종 점수
- 판단·confidence
- 모델·전략·버전 정보
- 내러티브·근거·경고·확인·무효화 조건 JSON
- fallback 정보
- 데이터 품질·기준시각·생성시각

JSON 필드는 §4.3의 최소 구조를 따른다. API는 저장된 문자열을 그대로 반환하지 않고 파싱·검증 후 envelope에 담는다.

유니크 기준:

```text
(scan_id, market, symbol)
```

- `id`
- `strategy_id`
- `market`
- `enabled`
- `automation_level` (`0`~`6`)
- `auto_approve`
- `auto_execute`
- `max_daily_orders`
- `max_daily_loss_pct`
- `max_risk_per_trade_pct`
- `max_position_pct`
- `max_market_exposure_pct`
- `min_final_score`
- `min_rule_score`
- `max_risk_score`
- `allow_fallback_trade`
- `allow_stale_data_trade`

### 6.8 `ai_stock_timing_signals`

2차 실시간 분석(§4.8)이 생성한 진입/청산 타이밍 신호를 1차 점수와 분리해 기록한다.

- `id`
- `strategy_id`
- `market`
- `candidate_id` (1차 후보 풀 참조)
- `symbol`
- `instrument_type` (`stock` | `etf`)
- `signal_type` (`entry` | `exit` | `hold`)
- `trigger` (`breakout` | `pullback` | `momentum` | `stop` | `target` | `invalidation` | `time`)
- `ref_price`
- `signal_price`
- `ai_timing_confidence`
- `decision` (`proceed` | `blocked`)
- `blocked_reason`
- `automation_level`
- `data_as_of`
- `created_at`

규칙:

- 신호는 1차 후보 풀에 있는 종목에 대해서만 생성한다(§4.8). `candidate_id` 없는 즉흥 신호를 만들지 않는다.
- 신호가 곧 주문이 아니다. `decision=proceed`여도 §1.2·§5.12 정책·리스크 게이트를 통과해야 주문으로 이어진다.
- 실시간 데이터 stale/단절 시 `entry` 신호를 생성하지 않고 보유 보호용 `exit`만 허용한다.

### 6.9 시황 브리핑 파일 (DB 아님)

월/주/일 시황(§4.7)은 DB가 아니라 파일로 저장한다(`.runtime/ai_stock/briefings/{market}/...`). DB에는 최신 브리핑의 경로·해시·기준시각만 참조로 남길 수 있고, 본문은 파일에서 읽는다. 런타임 산출물이므로 커밋하지 않는다.

## 7. API 공통 규칙

### 7.1 응답 envelope

```json
{
  "ok": true,
  "market": "KR",
  "data": {},
  "meta": {
    "data_as_of": "ISO-8601",
    "data_quality": "good",
    "stale": false,
    "fallback_used": false
  },
  "safety": {
    "dry_run": true,
    "trading_env": "demo",
    "enable_live_trading": false,
    "require_approval": true
  },
  "errors": []
}
```

### 7.2 HTTP 상태

| 상태 | 용도 |
|---|---|
| 400 | 입력값·schema 오류 |
| 404 | 후보·전략·계획 없음 |
| 409 | 중복 스캔, 잘못된 상태 전이, 안전 조건 미달 |
| 422 | FastAPI 요청 검증 오류 |
| 503 | 필수 시장 데이터 또는 AI provider 사용 불가 |

AI 실패가 결정적 점수 fallback으로 처리 가능하면 503 대신 정상 응답과 `fallback_used=true`를 반환한다.

### 7.3 전체 엔드포인트 목록

탭별 API(§5)에 더해 상태·설정·시황·유니버스·자동화·실시간 API를 포함한다. 모두 공통 envelope(§7.1)와 `market` 규칙을 따른다.

```text
# 상태·설정 (§1.4·F02·F41)
GET    /api/ai-stock/status                         # 안전 상태(DRY_RUN 등)·모델 상태·데이터 신선도
GET    /api/ai-stock/settings                        # AI스톡 설정 그룹 조회
PATCH  /api/ai-stock/settings                        # 설정 변경(범위 검증·감사 로그)

# 시황 브리핑 파일 (§4.7·§6.9)
GET    /api/ai-stock/briefings?market=&period=monthly|weekly|daily
GET    /api/ai-stock/briefings/{period}/{key}        # 특정 브리핑 본문(json/md)
POST   /api/ai-stock/briefings/generate?market=&period=   # 시황 파일 생성

# 종목 유니버스 (§4.6)
GET    /api/ai-stock/universe?market=                # 통과/제외 목록과 제외 사유

# 탭 1~9 (§5, 상세는 각 탭 참조)
GET    /api/ai-stock/overview · GET /narratives · POST /scans · GET /scans/{id}
GET    /candidates · GET /candidates/{id} · CRUD /watchlist
GET    /portfolio · GET /performance · GET /strategies · /execution-plans

# 자동화 정책·실행 이력 (§6.6·§6.7)
GET    /api/ai-stock/automation-policies?market=&strategy_id=
PUT    /api/ai-stock/automation-policies/{strategy_id}   # 정책 변경(검증·승인 무효화)
GET    /api/ai-stock/automation-runs?market=&strategy_id= # 실행 단계·차단 사유 이력

# 2차 실시간 타이밍 (§4.8·§6.8)
GET    /api/ai-stock/timing-signals?market=&candidate_id=
```

규칙:

- 정책 변경(`PUT automation-policies`)은 관련 검증·승인 상태를 무효화한다(§5.8).
- 설정·정책 변경은 안전 가드(환경변수 계층)를 우회할 수 없다(§1.4).
- 자동화·타이밍 엔드포인트는 주문을 직접 내지 않고 자동화 엔진·정책 게이트를 통해서만 집행으로 이어진다(§5.12).

### 8.1 진입 방식 및 공통 헤더

*   **진입 경로:** 신규 경로 `/ai-stock` 제공. 신규 파일 `ai_stock.html`, `ai_stock.js`, `ai_stock.css`로 기능을 격리하여 개발.

*   **공통 헤더 구성:**
    *   **조회 필터:** 시장 선택(전체 / 한국(AI한스톡) / 미국(AI미스톡) 드롭다운), 분석 기준일 달력, 전략 선택 드롭다운, 최소 종합 점수 및 최대 위험 점수 제어 슬라이더.
    *   **상태 표시 및 제어:** `[데이터 새로고침]` 버튼, `[AI 분석 실행]` 버튼, 최근 분석 기준시각 배지, AI 모델 상태 배지, 데이터 신선도 표시기, 현재 안전 모드(Dry Run 및 실거래 차단 정보) 상시 노출 영역.

### 8.1.1 UI 단계별 범위

UI는 안전 상태와 데이터 신뢰성 표시를 먼저 구현하고, 고급 시각 효과는 핵심 API와 저장 구조가 안정된 뒤 추가한다.

| 단계 | UI 범위 |
|---|---|
| 단계 0 | `/ai-stock` 진입점, 공통 헤더, safety 배지, 빈 상태, stale/fallback 배너 자리 |
| 단계 1 | 탭1·탭2 읽기 전용 목록/카드, 데이터 기준시각, `insufficient_data` 표시 |
| 단계 2 | 후보 표, 상세 패널, 점수 구성 표시, 스캔 진행 상태 |
| 단계 3 이후 | 관찰 상태 전이, 포트폴리오/성과/실행 계획 화면, 차트와 다운로드 |

어떤 단계에서도 색상·애니메이션이 stale, fallback, insufficient_data, safety 상태를 가리면 안 된다.

대시보드의 몰입감과 전문성을 위해 세련된 다크 모드를 기본으로 적용하고, 미세한 빛 효과(Glow)와 그라디언트를 조합하여 최고의 첫인상을 주도록 설계한다.

### 8.3 인터랙션 및 마이크로 애니메이션

*   **탭 전환 효과:** 세부 탭을 클릭하여 변경할 때, 화면이 딱딱하게 끊기지 않고 CSS 트랜지션을 사용하여 아래에서 위로 `translateY(8px)`만큼 밀려 올라오며 페이드인(`opacity: 0`에서 `1`로 0.2초)되는 모션 적용.
*   **호버 피드백:** 테이블의 종목 행이나 요약 카드에 마우스 오버 시, 은은한 스케일 업 효과(`transform: translateY(-2px)`)와 함께 보더 테두리에 얇은 그라디언트 네온 글로우가 적용되어 사용자의 주시 영역을 명확히 함.
*   **스켈레톤 로더 (Shimmering Loader):** 분석 실행이나 스캔 대기 중, 고정된 대기 문구 대신 콘텐츠 구조의 모양을 딴 회색 박스 위로 밝은 그라디언트 광원이 부드럽게 흘러가는 쉬머(Shimmer) 애니메이션을 탑재해 시각적 지루함 해소.
*   **애니메이션 최적화:** 브라우저 성능을 위해 하드웨어 가속이 가능한 CSS `transform`과 `opacity` 위주로 애니메이션을 작성하며, 시스템의 `prefers-reduced-motion` 설정을 탐지하여 애니메이션을 꺼주는 반응형 CSS 적용.

### 8.4 세부 탭별 프리미엄 디자인 구성

#### ① 탭 1 — 시장 브리핑
*   **국면 원형 대시보드:** KR/US의 시장 상태를 대형 원형 게이지(Radial Track)로 렌더링. 강세(Bullish)는 Emerald, 약세(Bearish)는 Rose 그라디언트 채우기가 적용된 게이지 바 디자인.
*   **지표 요약 카드:** 긍정/위험 요인 리스트를 명확한 카드 아이콘과 함께 투명 글라스 패널로 배치.
*   **시황 파일 뷰어(§4.7):** 월/주/일 AI 시황 브리핑을 탭/달력으로 선택해 열람하고, 신선도 배지와 "다음 기간 관전 포인트"를 카드로 표시. 파일이 stale이면 경고 배너를 띄운다.

#### ② 탭 2 — 내러티브
*   **내러티브 그리드 카드:** 각 테마별 내러티브 카드는 강도와 긍정/부정 감성에 맞춰 테두리 그라디언트가 다르게 발광(Glow)함.
*   **최신성 및 출처 배지:** 출처 다양성과 잔여 유효시간(TTL) 상태를 작은 캡슐 칩(Capsule Badge) 형태로 카드 우측 상단에 정렬하여 한눈에 식별 가능하게 함.

#### ③ 탭 3 — AI 종목 발굴
*   **스캔 시각화:** `[AI 분석 실행]` 시, Indigo 계열의 웨이브 프로그레스바가 흐르며 실시간 작업 상태 전달.
*   **스마트 데이터 테이블:**
    *   룰점수, 기술점수, AI점수 등 주요 필드는 셀 내부에 아주 얇은 파스텔 그라디언트 바 차트(In-cell Bar) 형태로 진행도를 오버레이하여 크기 비교가 바로 가능함.
    *   Confidence(신뢰도) 값은 원형 도트(Dot) 미터기로 채워 표기하고 툴팁으로 `"신뢰도(상승확률 아님)"`에 대한 정의 제공.
*   **유니버스 필터 칩(§4.6):** ETF/개별주 토글, 소형주 제외 기준(시총·거래대금) 표시, 제외 종목·사유를 별도 패널로 확인. (이 화면이 1차 배치 종목 선정 결과를 보여준다.)

#### ④ 탭 4 — 종목 분석
*   **5개 점수 다각형 레이더 차트:** 기존 Chart.js를 재사용해 5개 기여 점수(Rule, Technical, Momentum, Narrative, AI)를 한눈에 볼 수 있는 정형화된 레이더 차트를 중앙에 배치.
*   **AI 리포팅 말풍선:** AI의 종합 코멘트와 핵심 근거(Positive/Negative)는 마치 프리미엄 슬랙/메신저 풍선 같은 말풍선 스타일 박스로 정렬하여 텍스트 가독성을 최대로 높임.

#### ⑤ 탭 5 — 관찰종목
*   **상태 추적 리스트:** `Watch (관찰)` ➔ `Confirmed (검증완료)` ➔ `Rejected (기각)` 상태 전이를 그리드 카드 형태로 배열.
*   **확인 클릭 마이크로 애니메이션:** 검증 조건들을 통과하고 `Confirm` 상태 전이 버튼을 누르면 체크 박스가 세련되게 회전하며 활성화되는 녹색 스파크 피드백 모션 구현.

#### ⑥ 탭 6 — 포트폴리오
*   **비중 대조 차트:** 기존 Chart.js Donut 차트를 사용하여 보유량과 목표 비중을 반투명으로 겹쳐 그리는 프리미엄 레이아웃 적용.
*   **환율 보정 스위치:** KRW/USD 원통화가 섞여서 표기되지 않도록, 상단 우측에 세련된 슬라이딩 스위치(원통화 분리 표기 / KRW 환산 합산 / USD 환산 합산)를 제공.

#### ⑦ 탭 7 — 성과 검증
*   **백테스트 누적 성과 차트:** `Rule-Only` 성능 선과 `AI-Filtered` 성능 선을 뚜렷이 구분하여 라인 아래에 채우기 색상이 은은하게 감도는 Area Chart 구현. MDD 및 Sharpe Ratio 비교 수치를 대형 넘버 카드(Hero Metrics) 형태로 강조.

#### ⑧ 탭 8 — AI 전략 관리
*   **JSON 프로필 스키마 뷰어:** 전략 가중치와 하이퍼파라미터 코드를 볼 수 있도록 어두운 톤의 코드 구문 하이라이트 스타일(VS Code Monokai 테마풍) 적용.
*   **활성화 버튼:** 승인 상태 및 적용 스위치는 부드럽게 슬라이드되는 글로우 토글 스위치 사용.
*   **자동화 정책 패널(§6.6):** `automation_level`(0~6) 단계 슬라이더와 현재 상태, `auto_approve`·`auto_execute` 토글, 리스크 한도·유니버스 기준·임계값 편집(§1.4 설정). Level 5/6은 위험 상태와 전제조건(approved·실행 환경 안전 가드)을 강조 표시한다.
*   **스케줄·실행 이력:** 시장별 스케줄 설정과 디스패치 실행 이력(§5.8 패널)을 같은 화면에서 확인.

#### ⑨ 탭 9 — 실행 계획
*   **2차 실시간 타이밍 보드(§4.8·§5.9):** 1차 후보 풀의 실시간 진입/청산 신호를 종목별로 표시. `proceed`/`blocked`·`trigger`·타이밍 확신도·기준시각, KR 실시간/US 폴링 라벨 구분, 단절 시 "신규 진입 중단·청산만" 배너.
*   **다중 안전 조건 LED 위젯:** 일일 손실 한도 검사, 비중 체크, stop_price 유무 등 여러 조건을 체크리스트로 나열하고, 통과 시 녹색 LED 등(Glow)이 켜지는 세련된 체크리스트 패널 설계. 자동화 Level별 진행/차단 단계도 함께 표시.
*   **승인 전송 버튼:** 최종 승인 대기열로 계획을 전송하는 버튼은 마우스를 올릴 때 맥박이 뛰듯 은은하게 팽창/수축하는 펄스(Pulse) 애니메이션을 주어 최종 실행 단계의 주의력을 높임.

### 8.5 안전성 시각 정보 보장 원칙 (UI 원칙)

1.  **텍스트 상태 정보 필수 병행 표기 (색상 의존 탈피):**
    *   색상(빨강/초록)으로만 의미를 전달하는 것을 금지한다. 색약 사용자를 위해 모든 스코어 보드, 경고 패널, 판단 배지에는 점수와 함께 반드시 명시적인 문자 상태 라벨(`[92.5점 (강력 관찰)]`, `[15.0점 (위험 차단)]`, `[BULLISH (강세 국면)]`)을 `0.85rem` 크기 이상의 문자열 괄호 설명으로 강제 포함한다.
2.  **데이터 신선도 및 Fallback 탑 알림 배너:**
    *   조회하는 데이터가 해당 종류별 TTL(§9: 장중가격·일봉·시장국면·내러티브·AI평가별로 분리)을 초과한 Stale 상태이거나, 외부 모델 실패로 Rule 기반 점수로 자동 Fallback이 발생한 상태라면 화면 전체 최상단에 얇고 눈에 띄는 배너를 띄운다 (Amber/Rose 배경 그라디언트, 강제 경고 아이콘 탑재).
3.  **라이브러리 및 하드웨어 한계 준수:**
    *   외부 차트 라이브러리를 추가 임포트하지 않고, 기존에 사용 중인 Chart.js의 기능 범위를 초과하지 않도록 Canvas 요소를 활용하여 최적화한다.
    *   1차 개발 범위에서 무거운 드래그 앤 드롭 방식을 배제하고, 명시적인 클릭 스위치 버튼을 통한 상태 전이와 서버 확인 팝업을 우선 채택한다.

### 8.6 그래프·화면 사용 편의성 강화 (Usability)

시각 효과보다 **읽기 쉽고 다루기 쉬운 화면**을 우선한다. 아래 편의 기능은 전부 기존 Chart.js 범위와 표준 HTML/CSS/JS로 구현하며, 신규 라이브러리를 추가하지 않는다.

#### 8.6.1 그래프(차트) 사용 편의

*   **공통 차트 컨트롤 바:** 모든 차트 우상단에 기간 선택(`1D / 5D / 20D / 60D`), 범례 항목 클릭 토글(특정 시리즈 숨김/표시), 값 라벨 on/off, `PNG 저장`·`CSV 내려받기` 버튼을 표준 위치에 고정 배치한다.
*   **정밀 툴팁(Hover Tooltip):** 마우스(또는 터치) 위치의 모든 시리즈 값을 한 박스에 모아 보여주고, 기준시각·통화·단위를 함께 표기한다. AI한스톡은 `원/KRW`, AI미스톡은 `$/USD`로 단위를 분리 표기한다.
*   **비교 모드:** 성과 검증(탭7)·시장 브리핑(탭1)에서 `AI 순위 vs rule-only vs 벤치마크`처럼 여러 선을 겹쳐 보되, 토글로 한 번에 한 기준만 강조(나머지는 흐리게)할 수 있게 한다.
*   **읽기 보조선:** 0% 기준선, 임계선(최소 종합 점수·최대 위험 점수)을 점선 가이드로 표시해 합격/탈락 구간을 눈으로 바로 구분한다.
*   **빈/부족 데이터 처리:** 데이터가 없거나 stale이면 빈 캔버스를 방치하지 않고 "데이터 없음 / 기준시각 경과" 안내와 재조회 버튼을 차트 자리에 표시한다(허위 0 라인 금지).
*   **과밀 방지:** 항목이 많은 막대/도넛은 상위 N개만 표시하고 나머지는 `기타`로 묶으며, 전체 보기 토글을 제공한다.

#### 8.6.2 표·목록 사용 편의

*   **정렬·필터·검색:** 모든 표 헤더는 클릭 정렬(오름/내림), 종목명·코드 즉시 검색, 시장(AI한스톡/AI미스톡)·판단(strong_watch~avoid)·점수 구간 필터를 제공한다. 적용된 필터는 칩(chip)으로 보여주고 한 번에 초기화할 수 있다.
*   **고정 헤더·고정 열:** 표가 길어도 헤더 행과 첫 열(종목)을 스크롤 고정(sticky)해 항상 기준을 볼 수 있게 한다.
*   **밀도 토글:** `보통 / 촘촘(compact)` 행 높이 전환으로 한 화면에 보는 정보량을 사용자가 조절한다.
*   **인셀 시각화 + 숫자 병행:** 점수 셀의 인셀 막대는 보조 표현일 뿐, 숫자와 텍스트 상태 라벨(§8.5)을 항상 함께 표기한다.
*   **열 선택·기억:** 보고 싶은 열을 켜고 끌 수 있으며, 사용자의 정렬·필터·밀도·열 선택은 로컬 저장(localStorage)으로 다음 방문 시 유지한다.

#### 8.6.3 화면 탐색·조작 편의

*   **드릴다운 일관성:** 어디서든 종목 행을 클릭하면 우측 상세 패널(또는 모달)로 탭4 종목 분석을 열고, `ESC`로 닫는다. 목록 위치(스크롤·필터)는 보존한다.
*   **공통 헤더 고정:** 시장 필터·기준일·전략·점수 임계·`AI 분석 실행`을 담은 공통 헤더는 스크롤해도 상단 고정해 어느 탭에서든 즉시 재조회할 수 있게 한다.
*   **키보드 지원:** 탭 이동(`1`~`9` 또는 좌우 화살표), 목록 행 이동(↑/↓), 상세 열기(Enter), 닫기(ESC), 새로고침(`R`) 단축키를 제공한다.
*   **진행 상태 가시화:** 스캔·분석 실행 시 버튼을 로딩 상태로 잠그고, 스켈레톤 로더와 단계 텍스트(유니버스 구성→점수→AI평가)를 보여주되 가짜 퍼센트는 쓰지 않는다.
*   **상태 요약 스트립:** 각 탭 상단에 `후보 N · strong_watch N · stale N · fallback N`처럼 핵심 카운트를 한 줄로 요약해 전체 상황을 즉시 파악하게 한다.

#### 8.6.4 반응형·접근성

*   **반응형 레이아웃:** 데스크톱은 다단(목록+상세 패널) 그리드, 좁은 화면은 단일 컬럼으로 자동 전환하고 표는 가로 스크롤로 보존한다.
*   **터치 타깃:** 버튼·토글·칩은 최소 `40px` 터치 영역을 확보한다.
*   **접근성:** 색상 외 텍스트 병행(§8.5), 키보드 포커스 윤곽 표시, 차트에 대체 텍스트/요약 표 제공, `prefers-reduced-motion` 존중을 기본으로 한다.

> 이 편의 기능들은 데이터 신뢰성·안전 표시 원칙(§8.5)을 약화하지 않으며, 어떤 정렬·필터·표시 옵션도 stale·fallback·`insufficient_data` 경고를 숨기지 못한다.

## 9. 캐시와 데이터 신선도

캐시 키:

```text
market
symbol
data_as_of
feature_hash
narrative_hash
briefing_hash       # 시황 파일(월/주/일) 컨텍스트(§4.7)
universe_hash       # 유니버스 정책 스냅샷(§4.6)
strategy_id
strategy_version
profile_hash
provider
model
prompt_version
```

TTL은 설정 가능하게(§1.4) 두고 데이터 종류별로 분리한다.

- 장중 가격
- 일봉
- 시장 국면
- 내러티브
- AI 평가(1차)
- 시황 브리핑(월/주/일, `briefing_freshness_min`)
- 유니버스 스냅샷

원칙:

- 캐시 파일은 `.runtime/ai_stock/` 아래에 둔다.
- stale AI 캐시는 상세 조회 참고용으로만 사용 가능하다.
- stale 결과는 `confirmed`나 실행 계획 생성에 사용할 수 없다.
- **2차 실시간 타이밍 신호(§4.8)는 캐시하지 않는다.** 실시간성이 핵심이므로 매번 신선한 시세로 평가하고, 단절 시 신규 진입을 멈춘다.
- 시황 브리핑 파일이 신선도(`briefing_freshness_min`)를 넘으면 1차 점수화·자동 타이밍에 사용하지 않는다(§4.7).
- TTL 값과 데이터 기준시각을 응답 meta에 포함한다.

## 10.1 2명 에이전트 작업 분장

두 에이전트는 같은 파일을 동시에 크게 수정하지 않는다. Agent A는 Python/DB/API/자동화 엔진을 담당하고, Agent B는 dashboard 라우팅 연결, HTML/CSS/JS, 사용성, 문서와 화면 검증을 담당한다. 충돌 가능성이 큰 파일은 작업 순서를 고정한다.

| 구분 | Agent A — Backend/Automation | Agent B — Dashboard/UI/Docs |
|---|---|---|
| 주 책임 | 도메인 모델, DB, 서비스, API, 점수, AI, 리스크, 자동화 정책, 스케줄 | 화면, JS 상태관리, CSS, 탭 UI, 배너/표/차트, 라우터 등록 보조, 문서 |
| 먼저 작업 | F01~F06, F10, F13~F16 | F00, F02 UI 표시, F33 |
| 중간 작업 | F18, F20, F22, F24, F25, F27, F37, F38 | F17, F19, F21, F23, F26, F30 |
| 후반 작업 | F28, F29, F31, F39, backend 테스트 | F32, F34, F40, 문서, 브라우저 검증 |
| 테스트 책임 | Python unit/API tests, DB migration tests, policy tests | `node --check`, UI smoke, 브라우저 화면 검증, 접근성 기본 점검 |
| 금지 | UI에서 점수/안전 판정 재계산, 브로커 API 우회 호출 | 서버 검증 없이 confirmed/approval/order 상태 변경 |

### Agent A 상세 지시

1. `src/ai_stock/` 패키지와 `src/db/ai_stock_repository.py`를 만든다.
2. `repository.init_db()`에 AI스톡 테이블 helper를 연결하되 기존 DB를 파괴하지 않는다.
3. 모든 저장 함수에서 `market=ALL`을 거부한다.
4. `ai_stock_scans` 생성은 `queued/running` 중복을 차단하고 stale-running TTL을 처리한다.
5. `ModelPredictor`의 `probability`를 `confidence`로 직접 노출하지 않는다.
6. `automation_level`은 서버에서만 판정한다. UI 값은 요청 입력일 뿐 신뢰하지 않는다.
7. Level 5/6 자동화는 기존 시장별 승인·주문 경로만 사용한다. 신규 브로커 API 우회 경로를 만들지 않는다.
8. 외부 KIS/OpenAI/뉴스/미스톡 네트워크는 테스트에서 모두 mock한다.

### Agent B 상세 지시

1. `/ai-stock` 화면을 독립 파일로 만든다. 기존 `index.html`, `app.js`, `style.css`는 링크 추가 외에는 건드리지 않는다.
2. UI는 처음부터 9개 탭 shell을 제공하되, 미구현 탭은 명시적 빈 상태와 필요한 API명을 표시한다.
3. 모든 API 응답의 `safety`, `meta.stale`, `meta.fallback_used`, `errors`를 화면 상단에서 숨기지 않는다.
4. 색상만으로 상태를 표현하지 말고 텍스트 라벨을 병행한다.
5. UI에서 점수, confirmed, 자동 주문 허용 여부를 재계산하지 않는다. 서버 응답을 표시만 한다.
6. Level 5/6 자동화 UI는 위험 상태를 명확히 표시하고, 현재 정책과 차단 사유를 같이 보여준다.
7. Chart.js 외 신규 차트 프레임워크를 추가하지 않는다.

## 단계 7 — 스케줄 자동 실행 통합 및 무중단 검증

- `dispatch_due_schedules()`에 AI한스톡/AI미스톡 자동화 엔진 분기 추가(주문 경로 `run_scheduled_cycle` 직접 미사용, §5.12.2)
- AI스톡 `strategy_id` 격리 적용. Level 0~4는 주문 카테고리 없음, Level 5/6은 시장별 기존 승인·주문 경로만 허용
- 시장별 실행 윈도우(KR 마감 후 / `is_us_market_open()`)와 중복 실행 락 연결
- 성과 갱신 일일 잡 등록
- 무중단 시나리오 검증: AI키 없음·online 차단·종목 부분 실패에서 스캔이 fallback으로 완료되는지
- 스케줄이 전략별 `automation_level`을 초과해 진행하지 않는지 검증

완료 기준:

- 스케줄 실행이 정책 범위를 초과해 승인·주문을 자동 생성하지 않음
- 한 시장 실패가 다른 시장에 전파되지 않음
- AI 차단 시 룰 fallback으로 스캔 완료(중단 없음)
- 동일 시장 중복 스캔 차단

## 단계 8 — 2차 실시간 타이밍과 자동매매 집행

- 2차 실시간 분석 엔진(F39): 1차 후보 풀에 대한 실시간 재평가(§4.8)
- KR 실시간 시세 구독(`KISWebSocketClient`), US 폴링 주기 대체
- 진입/청산 타이밍 신호 생성 → `ai_stock_timing_signals`(§6.8)
- `automation_level`·리스크·정책 게이트 통과 시에만 기존 시장별 주문 경로로 자동매매(§1.2, §5.12)
- 실시간 데이터 stale/단절 시 신규 진입 중단, 보유 보호(청산·stop)만 유지
- demo 환경에서 우선 검증(§1.2.1)

완료 기준:

- 2차 신호는 1차 후보 풀 종목에 대해서만 생성됨(즉흥 매수 없음)
- 신호가 정책·리스크 게이트 없이 주문으로 이어지지 않음
- 실시간 단절 시 신규 진입 차단·청산만 허용
- 자동매매가 기존 시장별 승인·주문 경로만 사용(브로커 우회 없음)
- demo 검증 후에만 실계좌 전환(§1.2.1)

## 14. 착수 전 결정 또는 확인 항목

1. **US 내러티브 데이터**  
   현재 내러티브 수집·테마 맵이 US를 충분히 지원하는지 검증해야 한다. 미지원이면 1차 버전은 KR 내러티브만 활성화하고 US는 `insufficient_data`로 표시한다.

2. **AI provider (실측)**  
   기존 `ModelPredictor`(`src/strategy/predict.py`, `provider="openai_responses"`, `gpt-5-mini`, OpenAI Responses API)를 래핑하는 것을 기본으로 한다. 운영 활성화에는 `ai_strategy_enabled=true`와 `openai_api_key`가 필요하며, 미설정 시 룰 fallback으로 동작한다(§5.11). probability→ai_score/confidence 매핑 규약(§5.11.2)을 반드시 적용한다. 신규 provider 추상화는 실제 두 번째 provider가 필요할 때 확장한다.

3. **AI스톡 기준 DB**  
   공통 스캔·성과 비교를 위해 메인 DB 저장을 기본으로 한다. 미스톡 DB에는 AI스톡 이력을 중복 저장하지 않는다.

4. **시장 데이터 범위**  
   VIX, breadth, 외국인·기관 수급처럼 현재 소스가 보장하지 않는 지표는 필수 조건으로 두지 않는다.

5. **포트폴리오 기본 표시**  
   기본은 `LOCAL`로 두고 KRW/USD 원통화를 분리한다. 환산 합계는 사용자가 통화와 환율 기준시각을 선택한 경우에만 표시한다.

6. **스케줄 등록 방식 (실측)**  
   AI한스톡/AI미스톡을 `strategy_schedules`에 별도 `strategy_id`로 등록하고 `dispatch_due_schedules()`에 분기를 추가한다. `run_scheduled_cycle`/`trader.run`을 직접 타지 않고 AI스톡 자동화 엔진을 호출하며, Level 0~4는 주문 카테고리 없이 분석·계획까지만 수행하고 Level 5/6은 시장별 기존 승인·주문 경로만 사용한다(§5.12).

7. **US 승인 대기열 연결 수준**  
   `US` 실행 계획은 미스톡 기존 승인 큐로만 전송해야 한다(§5.9). 1차에서 이 연결을 충분히 검증하지 못하면 `US`는 계획 생성까지만 허용하고 승인 전송은 차단한다.

8. **DB migration 적용 위치**  
   AI스톡 테이블 생성은 `src/db/repository.py:init_db()` 흐름에 연결한다(§6.0). 별도 저장소 파일만 만들고 초기화에 연결하지 않는 구현은 완료로 보지 않는다.

9. **첫 PR 범위**  
   첫 구현 PR은 단계 0으로 제한한다. 즉 `/ai-stock` 진입점, 공통 envelope, safety 상태, 빈 상태 UI까지만 포함하고 스캔·AI 호출·승인 연결은 넣지 않는다.

10. **유니버스 기준 확정(§4.6)**  
   시총·일평균 거래대금·최소 주가 하한과 ETF 포함 범위, 소형주 정의(시총 기준 외 지수 편입 제한 여부)를 시장별로 확정한다. 계획서 수치는 기본 제안값이며 정책(§6.6)에서 변경한다.

11. **2차 실시간 범위(§4.8)**  
   KR은 `KISWebSocketClient` 실시간, US는 폴링/비활성으로 시작한다. 미국 정규장이 한국 야간이므로 무인 운영·알림 정책과 US 2차 활성 시점을 결정한다.

12. **시황 파일 주기·보관(§4.7)**  
   월/주/일 생성 시각·보관 기간·신선도(`briefing_freshness_min`)를 확정한다. 런타임 산출물이므로 커밋하지 않는다.

## 15. 제외 범위

다음은 1차 구현 범위에서 제외한다. 자동 승인·자동 주문 기능 자체는 설계에 포함하지만, 정책·리스크·감사 로그 없이 무조건 실행되는 형태는 허용하지 않는다.

- 정책 없는 AI 자동 주문
- 정책 없는 AI 자동 승인
- 브로커 API 직접 호출 우회 경로
- 자동 재학습·자동 전략 승격
- 뉴스 원문 저장
- 실시간 웹소켓 기반 AI 재평가
- 드래그 앤 드롭 상태 변경
- 새로운 차트 프레임워크 도입
- AI한스톡/AI미스톡(KR/US) 전략 DB의 즉시 통합
- 근거 없는 목표가격·기대수익률 생성
