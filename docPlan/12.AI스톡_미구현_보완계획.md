## 1. 검토 결론

현재 구현은 AI스톡의 패키지 구조, 9개 탭 shell, 공통 API envelope, 주요 DB 테이블, 후보 발굴·관찰·실행 계획의 기본 흐름을 제공한다.

다만 다음 핵심 기능은 완료되지 않았다.

- Level 6 자동 주문 실제 실행
- 제거하기로 한 paper 조건의 완전한 삭제
- AI probability와 confidence 분리
- 전략 승인·리스크·환경·시장시간 기반 자동화 게이트
- 실제 데이터 기준시각에 따른 stale 판정
- 문서 수준의 유니버스, 종목 분석, 포트폴리오, 성과 검증
- 일부 API와 UI 조작 기능
- 스케줄 실행 경계와 자동화 정책 적용

현재 상태는 개발용 골격 및 제한적인 Level 0~5 흐름으로 본다. Level 6 자동 주문과 운영 투입은 보완 완료 전까지 허용하지 않는다.

## 3. P0 — 자동 주문 및 안전 게이트 보완

### 3.1 Level 6 실제 주문 실행

#### 현재 문제

`automation_service.run_strategy()`는 execute 게이트 통과 시 기존 주문 경로를 호출하지 않고 `executed` 카운트만 증가시킨다.

#### 보완

- KR은 기존 한스톡 승인·실행 서비스와 `OrderRouter` 경로를 호출한다.
- US는 기존 미스톡 승인·주문 실행 경로를 호출한다.
- 브로커 API를 AI스톡 서비스에서 직접 호출하지 않는다.
- 실제 결과의 주문 ID, 브로커 주문 ID, 성공·실패 상태를 저장한다.
- 주문이 실행되지 않았으면 `executed`로 기록하지 않는다.

#### 대상 파일

- `src/ai_stock/automation_service.py`
- `src/approval_service.py`
- `src/execution_service.py`
- `src/strategy/router.py`
- `src/mistock/trader.py`
- `src/db/ai_stock_repository.py`

#### 완료 조건

- Level 6 게이트 통과 시 기존 시장별 주문 경로가 정확히 한 번 호출된다.
- 주문 실패가 `completed` 또는 `executed`로 기록되지 않는다.
- KR 주문이 미스톡 경로를 사용하거나 US 주문이 한스톡 경로를 사용하지 않는다.

### 3.2 paper 조건 완전 제거

#### 현재 문제

문서에서는 paper 요구사항을 제거했지만 구현에는 다음 항목이 남아 있다.

- `require_paper_passed` DB 컬럼
- 정책 저장 허용 필드
- `paper_passed_required` execute 차단 사유
- 관련 테스트 입력

#### 보완

- AI스톡 자동화 정책과 게이트에서 paper 조건을 제거한다.
- 기존 DB 컬럼은 호환성 때문에 즉시 삭제하지 않아도 되지만 읽거나 판단에 사용하지 않는다.
- 신규 DB 생성문과 정책 API에서는 필드를 제거한다.
- 기존 행의 값과 관계없이 Level 6 판단이 가능해야 한다.

#### 대상 파일

- `src/ai_stock/automation_service.py`
- `src/db/ai_stock_repository.py`
- `tests/test_ai_stock_automation.py`
- `tests/test_ai_stock_realtime.py`

### 3.3 전략 승인 상태 검증

#### 현재 문제

자동 주문 게이트가 실제 전략의 `status=approved` 여부를 검사하지 않는다.

#### 보완

- 시장별 전략 저장소에서 전략을 조회한다.
- KR·US 전략 시장 일치를 검증한다.
- 선택된 전략 ID·version·profile hash가 후보 및 계획과 일치해야 한다.
- `approved`가 아니거나 profile hash가 변경되었으면 계획·승인·주문을 차단한다.

#### 완료 조건

- draft, verified, backtested, review_required, retired 전략은 Level 6 차단
- approved 전략만 다음 안전 검사로 진행
- 정책 변경이나 profile 변경 후 기존 승인 상태 무효화

### 3.4 실행 환경 조건 분리

#### 보완 규칙

```text
demo 주문:
TRADING_ENV=demo
DRY_RUN=false
ENABLE_LIVE_TRADING=false

실계좌 주문:
TRADING_ENV=real
DRY_RUN=false
ENABLE_LIVE_TRADING=true
```

- `TRADING_ENV` 허용값은 `demo|real`로 고정한다.
- `live`, `prod` 등 비표준 값은 시작 시 차단한다.
- KR과 US는 각 시장별 설정을 사용한다.
- `online_access_blocked=true`이면 신규 주문을 차단한다.

### 3.5 자동 승인 안전 조건

- 정책의 `auto_approve=true`만으로 환경 안전 가드를 우회하지 못하게 한다.
- 기존 자동 승인 설정과 승인 서비스 정책을 함께 확인한다.
- Level 5는 승인 대기열 등록까지만 수행한다.
- Level 6도 기존 승인·주문 감사 흐름을 유지한다.

### 3.6 리스크 정책 실제 적용

현재 저장만 되고 사용되지 않는 다음 정책을 게이트에 연결한다.

- `enabled`
- `max_daily_orders`
- `max_daily_loss_pct`
- `max_risk_per_trade_pct`
- `max_position_pct`
- `max_market_exposure_pct`
- 종목·섹터·전략 집중도
- 중복 계획·승인·주문
- 시장 거래 가능 시간
- 현금 및 최소 주문금액
- kill switch

모든 차단 결과는 코드화된 `blocked_reason`으로 저장한다.

## 4. P0 — AI 점수와 confidence 수정

### 4.1 probability 직접 노출 제거

#### 현재 문제

현재 구현은 `ml_score=probability`를 다음처럼 사용한다.

```text
ai_score = probability × 100
confidence = probability
```

이는 11번 문서의 “probability를 confidence로 노출하지 않는다”는 규칙을 위반한다.

#### 보완

- probability는 AI 점수 계산 입력으로만 사용한다.
- confidence는 최소한 다음 요소로 별도 산출한다.
  - model status
  - 입력 데이터 품질
  - 필수 feature 충족률
  - 캐시 사용 여부
  - provider 응답 schema 유효성
- 별도 신뢰도 근거가 없으면 `confidence=null`로 반환한다.
- low confidence이면 AI 가중치를 0으로 만들고 나머지 가중치를 재정규화한다.

#### 대상 파일

- `src/ai_stock/ai_evaluator.py`
- `src/ai_stock/scoring.py`
- `tests/test_ai_stock_core.py`
- 신규 `tests/test_ai_stock_ai_mapping.py`

## 5. P0 — 데이터 신선도와 실시간 안전

### 5.1 실제 원천 기준시각 사용

#### 현재 문제

후보의 `data_as_of`가 가격 데이터 기준시각이 아니라 스캔 실행 시각으로 저장된다.

#### 보완

- 시장 데이터 provider가 `data_as_of`를 필수 반환하도록 계약을 변경한다.
- 가격, 일봉, 지수, 내러티브, 시황 파일의 기준시각을 각각 추적한다.
- 가장 오래된 필수 데이터 또는 종류별 판정을 후보 meta에 저장한다.
- 기준시각이 없으면 stale로 처리한다.

### 5.2 실시간 stale 판정

#### 현재 문제

실시간 가격만 존재하면 connected로 처리하고, 기준시각이 없으면 현재 시각을 대신 넣는다.

#### 보완

- 실시간 quote에 `data_as_of`를 필수로 요구한다.
- `intraday_price` TTL을 초과하면 신규 entry를 차단한다.
- 기준시각 누락도 disconnected/stale로 처리한다.
- stale 상태에서 임의 exit 신호를 생성하지 않는다.
- 보유 보호 주문은 기존 stop 상태와 실제 보유 여부를 확인한 후에만 처리한다.

### 5.3 `allow_stale_data_trade` 정리

11번 문서의 다른 안전 규칙과 맞추기 위해 stale 데이터의 confirmed·계획·신규 주문은 항상 차단한다.

- 정책 필드 및 UI에서 `allow_stale_data_trade`를 제거한다.
- fallback 허용 여부와 stale 허용 여부를 분리한다.
- fresh 원천 데이터로 계산된 rule fallback만 정책적으로 허용할 수 있다.

## 13. P1 — DB 무결성과 정책 검증

### 13.1 원자적 중복 스캔 방지

현재 활성 스캔 조회와 INSERT가 별도 연결이다.

- 단일 연결에서 `BEGIN IMMEDIATE`
- 활성 스캔 조회
- stale-running 정리
- 신규 scan INSERT
- commit

순서로 처리한다.

### 13.2 저장값 검증

- `market`: KR/US
- `automation_level`: 0~6
- 점수·confidence: 허용 범위
- 상태 enum
- 음수 리스크 한도 금지
- `auto_execute=true`이면 최소 Level 6
- `auto_approve=true`이면 최소 Level 5

DB CHECK 또는 서비스 계층 검증을 추가한다.

### 13.3 계획·승인 연결 갱신

- 승인 전송 시 execution plan의 `approval_market`, `approval_db`, `approval_id`, `approval_status` 갱신
- 주문 결과와 execution run 연결
- JSON 필드는 조회 시 파싱해 반환

## 14. P2 — API envelope와 오류 처리

### 현재 문제

대부분의 응답 meta가 실제 stale/fallback 상태를 반영하지 않고 기본값 `false`를 반환한다.

### 보완

- 서비스 결과에서 `data_as_of`, `data_quality`, `stale`, `fallback_used` 계산
- ALL 조회는 시장별 meta를 포함
- `MarketError`를 모든 라우트에서 일관되게 HTTP 400으로 변환
- 필수 데이터 부족과 provider 장애를 구분
- 정상 fallback은 200 + `fallback_used=true`
- 실제 서비스 장애는 503

## 16. P2 — 스케줄 및 실시간 운영

### 보완

- 스케줄의 mode·auto_approve·시장 설정을 AI스톡 엔진에 전달
- 정책 `enabled=false`이면 실행하지 않음
- KR·US 시장시간 분리
- 2차 실시간 기능 활성화 여부와 주기 적용
- 동일 시장·전략 중복 실행 원자적 차단
- AI키 없음·온라인 차단·종목 부분 실패 격리
- 성과 갱신 잡 별도 등록
- 한 시장 실패가 다른 시장에 전파되지 않음

KR WebSocket 연결은 실제 구독 lifecycle, 재연결, stale 감지, 종료 처리를 구현한다. US 폴링은 시장시간과 `realtime_poll_seconds`를 적용한다.
