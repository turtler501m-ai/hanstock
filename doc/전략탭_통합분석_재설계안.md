# 전략 탭 통합 분석 및 재설계안

작성일: 2026-06-01

## 1. 목적

이 문서는 현재 대시보드의 `전략` 관련 화면 전체를 다시 정리하고, 기존 기능을 유지하면서 AI 전략 설정, 후보 탐색, 보유 종목 진단, 포트폴리오 배분, 실행 계획, 승인 대기열까지 하나의 일관된 흐름으로 재설계하기 위한 분석/설계안이다.

대상 화면:

- `신호/계획` 탭: 보유 종목 매매 신호, 신규 매수 후보, 후보 히스토리, 실행 계획
- `AI` 탭: AI 목표 비중 가이드, 포트폴리오 최적화
- `AI 전략` 탭: AI 전략 목록, 선택, 검증, 신규 전략 등록
- `주문승인` 탭: 후보/계획/AI 배분에서 생성된 승인 대기 주문 처리

관련 주요 파일:

- `web/templates/index.html`
- `web/static/js/app.js`
- `src/dashboard/core.py`
- `src/dashboard/routes/stock.py`
- `src/dashboard/routes/account.py`
- `src/db/repository.py`
- `src/strategy/seven_split.py`
- `src/strategy/predict.py`
- `src/strategy/allocator.py`

## 2. 현재 전략 탭 구성

### 2.1 보유 종목 매매 신호

화면 위치:

- `web/templates/index.html`: `dashboard-tab-strategy`
- `web/static/js/app.js`: `renderSignals()`
- API: `GET /api/signals`
- 서버 구현: `src/dashboard/core.py:get_signals()`, `build_dashboard_signals()`
- 전략 엔진: `src/strategy/seven_split.py:generate_signal()`

현재 역할:

- KIS 계좌의 보유 종목을 조회한다.
- 각 보유 종목의 일봉 데이터를 가져온다.
- RSI, RSI2, MACD, SMA, 볼린저 밴드, 수익률, 세븐 스플릿 룰을 기반으로 `buy`, `sell`, `hold` 신호를 계산한다.
- 화면에는 종목, 판단, 점수, 수량, RSI/RSI2, MACD, 근거를 표시한다.

현재 특징:

- 보유 종목 중심의 “진단” 기능이다.
- 실제 주문 생성은 바로 하지 않고, 사용자가 승인 대기열에 넣는 별도 흐름과 연결된다.
- AI 전략 선택값은 이 보유 종목 신호에는 직접 반영되지 않는다.

### 2.2 신규 매수 후보

화면 위치:

- `web/templates/index.html`: `panel-candidates`
- `web/static/js/app.js`: `renderCandidates()`, `buildCandidateStrategyMarkup()`
- API: `GET /api/candidates`
- 서버 구현: `src/dashboard/core.py:get_candidates()`, `build_dashboard_candidates()`
- 전략 엔진: `src/strategy/seven_split.py:find_candidates()`, `build_orders()`

현재 역할:

- 관심종목과 거래량 상위 종목에서 매수 후보를 찾는다.
- `select-ai-ranker`의 랭커와 `select-portfolio-optimizer`의 배분 방식을 API 파라미터로 전달한다.
- 룰 점수, AI 점수, 최종 점수, AI 모델 상태, fallback 사유를 후보 테이블에 표시한다.
- 후보별 예상 수량과 예상 금액을 계산한다.
- 후보 주문은 `승인대기` 버튼으로 승인 대기열에 넣을 수 있다.

현재 지원 랭커:

- GPT-5-mini 기반 추론 랭커
- 기본 기술 룰베이스 랭커
- AI 전략 탭에서 선택된 사용자 등록 전략

현재 지원 배분 방식:

- `score_tilted_inverse_vol`: 변동성 역수와 점수 기반 비중 배분
- `score_proportional`: 단순 점수 비례 배분
- `llm_confidence_weight`: AI/ML 확신도 기반 비중 배분

현재 특징:

- AI 전략 설정이 실제로 가장 많이 반영되는 곳이다.
- 후보 결과는 `scanned_candidates` DB에 저장된다.
- 후보 캐시가 존재하며, ranker/optimizer/AI signature가 캐시 유효성에 일부 반영된다.

### 2.3 매수 후보 포착 히스토리

화면 위치:

- `web/templates/index.html`: `panel-candidates-history`
- `web/static/js/app.js`: `renderCandidateHistory()`
- API: `GET /api/candidates/history`, `DELETE /api/candidates/history/{candidate_id}`
- 저장소: `src/db/repository.py`

현재 역할:

- 과거 후보 스캔 결과를 DB에서 읽어 표시한다.
- 히스토리 항목 삭제를 지원한다.

현재 한계:

- 어떤 AI 전략, 랭커, optimizer로 생성된 후보인지 충분히 저장하지 않는다.
- 룰 점수, AI 점수, 최종 점수, fallback 사유가 히스토리에 충분히 남지 않는다.
- 전략 성과 비교나 전략별 후보 품질 분석으로 이어지기 어렵다.

### 2.4 실행 계획

화면 위치:

- `web/templates/index.html`: `panel-execution-plan`
- API: `GET /api/execution-plan`
- 서버 구현: `src/dashboard/core.py:get_execution_plan()`

현재 역할:

- 다음 자동매매 사이클에서 실행할 수 있는 주문 계획을 보여준다.
- 주문 계획 항목은 승인 대기열로 보낼 수 있다.

현재 한계:

- 후보 탐색, 보유 신호, AI 배분 결과와의 관계가 화면상 명확히 분리되어 있다.
- 어떤 전략 판단이 실행 계획에 포함되었는지 추적성이 약하다.

### 2.5 AI 목표 비중 가이드

화면 위치:

- `web/templates/index.html`: `dashboard-tab-ai`, `table-ai-allocation`
- `web/static/js/app.js`: `renderAiAllocation()`
- API: `GET /api/ai-allocation`
- 서버 구현: `src/dashboard/routes/stock.py:get_ai_allocation()`
- 전략 엔진: `src/strategy/seven_split.py:generate_ai_weight_plan()`

현재 역할:

- 보유 종목을 대상으로 목표 비중을 계산한다.
- 보유 비중과 목표 비중 차이를 계산해 `buy`, `sell`, `hold` 액션을 제안한다.
- AI 전략 상세 근거 모달을 제공한다.
- 승인 대기열로 주문을 넘길 수 있다.

현재 특징:

- PPO 모델 파일이 있으면 강화학습 모델을 사용하려고 시도한다.
- 모델이 없으면 점수/변동성 기반 heuristic으로 fallback한다.
- 화면상 “AI”로 보이지만 실제 모델 사용 여부가 명확히 표시되어야 한다.

### 2.6 포트폴리오 최적화

화면 위치:

- `web/templates/index.html`: `panel-optimizer`
- API: `GET /api/portfolio-optimizer`
- 서버 구현: `src/dashboard/routes/account.py:get_portfolio_optimizer()`
- 전략 엔진: `src/strategy/seven_split.py:generate_portfolio_optimizer_plan()`

현재 역할:

- 보유 종목의 점수와 변동성을 사용해 목표 비중을 제안한다.
- 과도한 쏠림을 줄이는 보조 배분 기능이다.

현재 한계:

- AI 목표 비중 가이드와 목적이 일부 겹친다.
- 사용자가 어떤 상황에서 AI 배분과 포트폴리오 최적화 중 무엇을 봐야 하는지 명확하지 않다.

### 2.7 AI 전략 관리

화면 위치:

- `web/templates/index.html`: `dashboard-tab-ai-strategies`
- `web/static/js/app.js`: `renderAiStrategies()`, `syncStrategiesToDropdown()`
- API: `/api/ai-strategies`
- 저장소: `src/db/repository.py:load_ai_strategies()`, `save_ai_strategies()`

현재 역할:

- AI 전략 목록을 조회한다.
- 전략을 선택한다.
- 전략을 검증한다.
- 신규 전략을 등록한다.
- 선택된 전략을 `신호/계획` 탭의 후보 선정 랭커로 노출한다.

현재 한계:

- 여러 전략을 선택할 수 있지만 실제 적용은 첫 번째 선택 전략 중심이다.
- 전략 설명이 OpenAI 프롬프트에 직접 반영되지 않는다.
- LightGBM/PPO 같은 로컬 모델 전략은 이름상 존재하지만 검증과 실행 경로가 명확하지 않다.
- 전략 삭제/수정 기능이 없다.

## 3. 현재 데이터 흐름

### 3.1 보유 종목 신호 흐름

```text
사용자: 진단하기 클릭
→ GET /api/signals
→ KIS 잔고 조회
→ 보유 종목별 KIS 일봉 조회
→ generate_signal()
→ RSI/MACD/SMA/볼린저/수익률 기반 buy/sell/hold
→ 신호 테이블 표시
→ 필요 시 승인 대기열 등록
```

### 3.2 신규 매수 후보 흐름

```text
사용자: 랭커/optimizer 선택 후 찾기 클릭
→ GET /api/candidates?ranker=...&optimizer=...
→ KIS 잔고 조회
→ 관심종목 + 스캔 유니버스 구성
→ find_candidates()
→ 룰 점수 계산
→ AI 활성 시 상위 후보 OpenAI 점수 결합
→ build_orders()로 수량/예상금액 계산
→ 후보 결과 DB 저장
→ 후보 테이블 표시
→ 승인 대기열 등록 가능
```

### 3.3 AI 전략 적용 흐름

```text
사용자: AI 전략 탭에서 전략 선택
→ POST /api/ai-strategies/{id}/select
→ 선택 상태 DB 저장
→ syncStrategiesToDropdown()
→ 신호/계획 탭 select-ai-ranker에 노출
→ 후보 찾기 실행 시 해당 전략의 model/weight 사용
```

### 3.4 AI 목표 비중 흐름

```text
사용자: AI 탭에서 계산 클릭
→ GET /api/ai-allocation
→ KIS 잔고 조회
→ 보유 종목 일봉 조회
→ generate_ai_weight_plan()
→ PPO 모델 있으면 사용, 없으면 heuristic fallback
→ 목표 비중/차이/액션 표시
→ 승인 대기열 등록 가능
```

## 4. 핵심 문제 정의

### 4.1 전략 관련 화면이 분산되어 있음

현재 사용자는 다음 화면을 오가야 한다.

- AI 전략 탭에서 전략 선택
- 신호/계획 탭에서 후보 찾기
- AI 탭에서 목표 비중 계산
- 주문승인 탭에서 승인 처리

각 화면이 기능적으로 연결되어 있지만, 화면상 하나의 “전략 실행 파이프라인”으로 보이지 않는다.

### 4.2 전략 선택과 전략 실행의 의미가 모호함

AI 전략 탭에서 선택한 전략은 신규 매수 후보 랭커에만 주로 반영된다.

하지만 사용자는 선택 전략이 다음 영역에도 영향을 준다고 기대할 수 있다.

- 보유 종목 매매 신호
- AI 목표 비중
- 포트폴리오 최적화
- 실행 계획

현재는 이 영향 범위가 명확하지 않다.

### 4.3 AI 전략과 룰 전략의 역할 구분이 부족함

현재 구조상 AI는 룰 점수에 가중치로 결합되는 보조 점수에 가깝다.

반면 UI는 독립 전략, 로컬 모델, PPO, LightGBM, 배분기까지 모두 “AI 전략”처럼 보여준다.

### 4.4 검증 결과와 운영 상태가 부족함

전략은 다음 상태를 가져야 하지만 현재는 `selected` 정도만 있다.

- 등록됨
- 정적 검증 완료
- 백테스트 완료
- 모의운용 중
- 실전 적용 승인됨
- 비활성화/폐기됨

### 4.5 후보 히스토리와 전략 이력이 분리되어 있음

후보가 저장되지만 해당 후보를 만든 전략 설정이 충분히 남지 않는다.

나중에 “어떤 전략이 어떤 후보를 얼마나 잘 골랐는지” 분석하기 어렵다.

### 4.6 캐시와 설정 변경의 정합성 위험

전략 설명, profile, 검증 상태, 후보 승격 정책 등이 추가되면 기존 후보 캐시가 잘못 재사용될 수 있다.

캐시 키는 전략 버전과 profile hash까지 포함해야 한다.

## 5. 재설계 목표

전체 전략 탭은 다음 5단계 흐름으로 재정리한다.

```text
1. 전략 선택/검증
2. 보유 종목 진단
3. 신규 후보 탐색
4. 포트폴리오/비중 계획
5. 승인 대기열 생성 및 실행 추적
```

설계 원칙:

- AI는 주문 실행자가 아니라 점수/근거/비중 제안자로 제한한다.
- 실전 주문은 승인 대기열과 안전 설정을 반드시 거친다.
- 모든 후보와 주문 계획에는 사용 전략 id/version/profile hash를 남긴다.
- 전략은 상태값을 갖고, 검증되지 않은 전략은 실전 기본값으로 쓸 수 없게 한다.
- 기존 룰 기반 결과는 항상 fallback으로 유지한다.

## 6. 제안 화면 구조

### 6.1 전략 탭을 “전략 실행센터”로 재구성

기존 `신호/계획` 탭을 다음 섹션으로 정리한다.

1. 전략 선택 바
2. 보유 종목 진단
3. 신규 매수 후보
4. 후보 히스토리
5. 실행 계획

상단 전략 선택 바에는 다음 정보를 표시한다.

- 현재 적용 전략
- 전략 상태: draft/verified/paper_running/approved/retired
- 랭커 모델
- AI 가중치
- 배분 방식
- fallback 상태
- 마지막 검증 시각

### 6.2 AI 전략 탭은 “전략 관리/검증”으로 역할 고정

AI 전략 탭은 후보 탐색 화면이 아니라 전략 관리 화면으로 명확히 한다.

제공 기능:

- 전략 생성
- 전략 수정
- 전략 삭제
- 정적 검증
- 백테스트 검증
- 모의운용 시작/종료
- 기본 전략으로 승인
- 전략 이벤트 이력 보기

### 6.3 AI 탭은 “비중/배분 분석”으로 정리

AI 목표 비중 가이드와 포트폴리오 최적화는 목적이 비슷하므로 다음처럼 정리한다.

- `AI 목표 비중`: 현재 전략 기반 보유 종목 리밸런싱
- `포트폴리오 최적화`: 룰/변동성 기반 대체 배분안
- 두 결과를 비교하는 요약 영역 추가

## 7. 전략 데이터 모델 제안

### 7.1 ai_strategies 확장

현재 컬럼에 다음을 추가한다.

```text
status TEXT DEFAULT 'draft'
profile_json TEXT
strategy_version INTEGER DEFAULT 1
profile_hash TEXT
last_verified_at TEXT
last_backtested_at TEXT
last_paper_started_at TEXT
last_paper_completed_at TEXT
last_used_at TEXT
last_validation_result TEXT
```

### 7.2 strategy profile JSON

전략의 실제 동작 설정은 `profile_json`에 저장한다.

예시:

```json
{
  "strategy_type": "rebound",
  "risk_level": "balanced",
```

## 9. 전략별 적용 범위 정리

전략 설정이 어느 기능에 반영되는지 명확히 정의한다.

| 기능 | 전략 반영 여부 | 반영 항목 |
| --- | --- | --- |
| 보유 종목 신호 | 1단계에서는 미반영 | 기존 세븐 스플릿 룰 유지 |
| 신규 매수 후보 | 반영 | model, weight, profile, confidence, 후보 승격 정책 |
| 후보 주문 수량 | 반영 | optimizer, risk setting, max position |
| AI 목표 비중 | 2단계에서 반영 | risk_level, target allocation profile |
| 포트폴리오 최적화 | 2단계에서 반영 | optimizer profile |
| 실행 계획 | 반영 | strategy id/version 기록 |
| 승인 대기열 | 반영 | source, strategy id/version, fallback 상태 표시 |

권장 단계:

- 1단계: 신규 매수 후보에만 전략 profile 적용
- 2단계: AI 목표 비중과 포트폴리오 최적화에 적용
- 3단계: 보유 종목 매매 신호에도 선택 전략을 선택적으로 반영

## 11. 안전 정책

기본 유지값:

```text
DRY_RUN=true
TRADING_ENV=demo
ENABLE_LIVE_TRADING=false
REQUIRE_APPROVAL=true
```

추가 정책:

- `status != approved` 전략은 실전 기본 전략으로 지정 불가
- `backtest_pass=false` 전략은 후보 찾기 가능하지만 주문 승인 전 경고
- `ai_fallback_reason`이 있으면 주문 대기열에 fallback badge 표시
- `ENABLE_LIVE_TRADING=true`일 때 전략 변경/승인 시 추가 확인 필요
- AI 후보 승격은 `DRY_RUN=true`에서 먼저 허용
- 일일 AI 주문 후보 수 제한

## 14. 최종 권장안

가장 먼저 적용할 현실적인 설계는 다음이다.

1. AI 전략은 다중 선택이 아니라 단일 active strategy로 정리한다.
2. 전략에는 `status`, `profile_json`, `strategy_version`, `profile_hash`를 추가한다.
3. 신규 매수 후보부터 `strategy_id` 기반으로 재연결한다.
4. 후보 히스토리와 승인 대기열에 전략 id/version/source를 남긴다.
5. 전략 설명/profile을 OpenAI 프롬프트와 캐시 키에 반영한다.
6. 검증되지 않은 전략은 실전 기본 전략으로 승격할 수 없게 한다.
7. 여러 전략 적용은 즉시 앙상블이 아니라 비교/토너먼트 화면으로 구현한다.

이 방향이면 현재 기능을 크게 깨지 않으면서도, 전략 탭 전체가 “보이는 대시보드”에서 “검증 가능한 전략 운영 흐름”으로 확장된다.
