## 2. 적용 결론

내러티브 모멘텀 전략은 한스톡에 바로 실거래 전략으로 붙이기보다, 먼저 **AI 전략 후보 스캐너**로 붙이는 것이 적절합니다.

권장 적용 방식:

1. `src/strategy/narrative_momentum.py`에 순수 점수 계산 엔진을 만든다.
2. `config/theme_map.json`으로 테마-종목 매핑을 관리한다.
3. `.runtime/narrative_history.json`에 당일 내러티브 JSON을 저장한다.
4. 대시보드와 `/api/candidates` 계열에서는 후보 점수와 근거만 보여준다.
5. 주문은 기존 한스톡 원칙대로 `REQUIRE_APPROVAL=true` 승인 대기열을 거친다.
6. 최소 20~60거래일 후보 이력과 1/5/20일 forward return을 축적한 뒤 자동 주문 비중을 결정한다.

즉, 초기 버전의 목표는 "자동매수"가 아니라 **뉴스/공시 기반 테마 쏠림 후보를 구조화해서 기존 Seven Split, 급락반등, 하이킨아시 후보와 비교 가능한 점수로 만드는 것**입니다.

## 3. H1/H2 분석 요약

### H1 핵심

`H1.전략10.md`는 내러티브 모멘텀의 계산 규칙을 가장 직접적으로 정의합니다.

핵심 입력:

- `narrative_history`: 최근 3일간 내러티브 분석 JSON
- `theme_map`: 테마명과 종목 목록 매핑

핵심 필터:

```text
direction == "rising"
strength > 60
```

핵심 점수식:

```text
nar_score =
    strength
  * (1 + delta / 100)
  * day_weight
  * sentiment_mult
  * acceleration
  + shift_bonus
```

핵심 안전장치:

- 최신 내러티브 날짜가 오늘이 아니면 빈 결과 반환
- 점수는 수익률 예측값이 아니라 상대 순위 점수
- `theme_map` 품질이 낮으면 신호 누락 또는 오매칭이 발생

### H2 핵심

`H2.내러티브_모멘텀_유튜브_분석_보고서.md`는 영상 내용을 한스톡에 연결하는 초안입니다.

유용한 내용:

- 뉴스/공시 수집 -> LLM 구조화 -> 테마 매핑 -> 종목 점수화 -> 승인 대기 주문 흐름
- `src/strategy/narrative_momentum.py`, `config/theme_map.json`, `.runtime/narrative_history.json`, `tests/test_narrative_momentum.py` 배치 제안
- HD현대에너지솔루션 같은 테마성 사례를 통해 전략의 사용 장면 설명

보완할 점:

- 영상 성과 표현은 실제 한스톡 데이터로 검증 전까지 운영 근거로 쓰면 안 됩니다.
- YouTube 강의의 예시와 한스톡의 현재 종목 유니버스, 주문 승인, 리스크 정책을 분리해야 합니다.
- `H1` 부록 코드는 문서 예시 수준이며, 그대로 복사하면 Python 문법 공백 문제가 있습니다. 구현 시 새 모듈로 재작성해야 합니다.

## 6. 데이터 스키마

### 6-1. 내러티브 히스토리

저장 위치:

```text
.runtime/narrative_history.json
```

형식:

```json
[
  {
    "date": "2026-06-18",
    "market_mood": "risk_on",
    "mood_score": 72,
    "dominant_narratives": [
      {
        "theme": "AI 반도체 투자 확대",
        "strength": 88,
        "sentiment": "bullish",
        "direction": "rising",
        "affected_sectors": ["반도체", "AI", "전력인프라"],
        "key_facts": ["주요 기업 투자 확대", "HBM 수요 지속"]
      }
    ],
    "narrative_shifts": [
      {
        "theme": "AI 반도체 투자 확대",
        "change": "rising"
      }
    ]
  }
]
```

운영 규칙:

- 최신순 또는 날짜 포함 리스트를 허용하되, 엔진에서 날짜 기준 정렬합니다.
- 최근 3거래일만 점수에 사용합니다.
- `date`는 캘린더 날짜보다 **한국 거래일 기준**으로 해석해야 합니다.
- 휴장일에는 직전 거래일 데이터를 당일 신호로 오인하지 않도록 별도 `trading_date` 검사를 둡니다.

## 7. 구현 단계

### 1단계: 순수 엔진 구현

파일:

```text
src/strategy/narrative_momentum.py
```

주요 클래스/함수:

```python
class NarrativeMomentumStrategy:
    def calculate_signals(
        self,
        narrative_history: list[dict],
        theme_map: dict[str, list[dict]],
        today_str: str | None = None,
    ) -> list[dict]:
        ...
```

필수 기능:

- 날짜 정렬
- stale guard
- `direction == "rising"` 필터
- `strength > STRENGTH_MIN` 필터
- 전일 대비 `delta` 계산
- 2일/3일 연속 rising 가속도 계산
- sentiment 배율
- shift bonus
- 테마/섹터 fuzzy matching
- `0~100` 최종 점수 clamp

권장 출력:

```json
{
  "ticker": "000660",
  "name": "SK하이닉스",
  "score": 91.2,
  "final_score": 91.2,
  "rule_score": 91.2,
  "ml_score": null,
  "current_price": 0,
  "themes": ["반도체"],
  "reasons": [
    "내러티브: AI 반도체 투자 확대 strength=88 sentiment=bullish direction=rising"
  ],
  "strategy_id": "narrative_momentum_strategy"
}
```

`current_price`는 엔진에서 조회하지 않습니다. 가격 조회는 기존 후보/주문 생성 흐름에서 처리합니다.

### 2단계: 파일 로더와 CLI 도구

파일:

```text
tools/run-narrative-momentum.py
```

역할:

- `.runtime/narrative_history.json` 읽기
- `config/theme_map.json` 읽기
- 엔진 실행
- `.runtime/narrative_momentum_latest.json` 저장
- 상위 후보 콘솔 출력

이 단계는 대시보드나 주문과 분리해서 점수 계산만 빠르게 검증하기 위한 것입니다.

### 3단계: 테스트 추가

파일:

```text
tests/test_narrative_momentum.py
```

필수 테스트:

- 강한 rising 테마가 매핑 종목으로 점수화된다.
- `strength <= 60`은 제외된다.
- `direction != rising`은 제외된다.
- stale 날짜면 빈 결과를 반환한다.
- 2일/3일 연속 rising이면 가속도 배율이 붙는다.
- `theme`과 `affected_sectors` 둘 다 매핑 후보로 사용된다.
- 한 종목이 여러 테마에 포함되면 점수가 누적된다.
- fuzzy threshold를 높이면 오매칭이 줄어든다.

검증 명령:

```powershell
python -m unittest tests.test_narrative_momentum
python -m unittest discover -s tests
```

#### 전용 탭 구성

`/narrative-momentum` 화면은 다음 4개 탭으로 나눕니다.

| 탭 | 목적 | 주요 데이터 |
| --- | --- | --- |
| `요약` | 오늘 데이터 신선도와 전체 상태 확인 | 최신 날짜, 시장 분위기, 후보 수, stale 여부 |
| `시그널` | 종목별 내러티브 점수 확인 | ticker, name, score, themes, narratives, approval action |
| `내러티브 이력` | 최근 3~20거래일 테마 흐름 확인 | date, theme, strength, sentiment, direction |
| `테마 매핑` | theme_map 품질 확인 | theme, mapped stocks, alias, 매칭 실패 테마 |

#### 요약 탭

표시 항목:

- 최신 내러티브 날짜
- 오늘 거래일과 일치 여부
- `market_mood`, `mood_score`
- 분석된 내러티브 수
- 매핑 성공 테마 수
- 매핑 실패 테마 수
- 생성된 후보 수
- 승인 대기열 생성 가능 여부

상태 배지:

```text
fresh       최신 거래일 데이터
stale       오래된 데이터
missing     narrative_history 없음
invalid     JSON schema 오류
blocked     ONLINE_ACCESS_BLOCKED 또는 안전조건 차단
```

#### 시그널 탭

컬럼:

| 컬럼 | 설명 |
| --- | --- |
| 순위 | 점수 내림차순 |
| 종목 | 종목명/티커 |
| 점수 | `final_score` |
| 테마 | 매칭된 theme_map key |
| 근거 | strength, sentiment, direction, acceleration |
| 상태 | fresh/stale, 가격조회 여부, 중복 승인 여부 |
| 액션 | 승인요청 버튼 |

승인요청 버튼 정책:

- stale 상태면 비활성화
- 점수 75 미만이면 비활성화
- 가격 조회 실패면 비활성화
- 이미 pending buy approval이 있으면 비활성화
- 클릭 시 `POST /api/narrative-momentum/queue-approval`

요청 payload:

```json
{
  "ticker": "000660",
  "name": "SK하이닉스",
  "score": 91.2,
  "price": 0,
  "reason": "내러티브 모멘텀 91.2점: AI 반도체 투자 확대",
  "strategy_id": "narrative_momentum_strategy"
}
```

#### 내러티브 이력 탭

목적은 "오늘 왜 이 종목이 떴는지"보다 "테마가 며칠째 강해지는지"를 보는 것입니다.

표시 방식:

- 날짜별 카드가 아니라 조밀한 테이블
- 테마별 strength 변화
- sentiment/direction 변화
- `new`, `rising`, `fading`, `disappeared` shift 표시
- 최근 3거래일은 점수 계산 대상임을 강조

권장 컬럼:

```text
date | theme | strength | delta | sentiment | direction | shift | affected_sectors | key_facts
```

#### 테마 매핑 탭

목적은 운영자가 `theme_map.json` 품질을 확인하는 것입니다.

표시 항목:

- 테마명
- 연결 종목 수
- 연결 종목 목록
- 최근 매칭된 내러티브
- 최근 매칭 실패 내러티브
- fuzzy score

초기 버전에서는 대시보드에서 직접 수정하지 않고 읽기 전용으로 둡니다. 수정은 `config/theme_map.json` 파일에서 하고, 화면에는 `theme-map/reload` 버튼만 둡니다.

#### 화면 디자인 원칙

- 기존 `index.html`의 대형 통합 탭에 직접 HTML을 더 붙이지 않습니다.
- `narrative_momentum.html`은 독립 페이지로 구성합니다.
- 공통 네비게이션에는 `내러티브` 버튼만 추가합니다.
- 카드 중첩을 피하고, 표와 상태 배지를 중심으로 구성합니다.
- 점수는 수익률 예측처럼 보이지 않도록 `점수` 옆에 `상대 순위` 문구를 작게 표시합니다.
- 실거래 버튼은 `매수`가 아니라 `승인요청`으로 표기합니다.

#### 프론트엔드 책임 분리

```text
web/static/js/narrative_momentum.js
```

담당:

- 탭 전환
- API fetch
- 테이블 렌더링
- stale/invalid 상태에 따른 버튼 비활성화
- 승인요청 POST
- 오류 메시지 표시

하지 않을 것:

- 점수 계산
- theme fuzzy matching
- 주문 수량 결정
- 실주문 실행

#### 백엔드 책임 분리

```text
src/dashboard/routes/narrative_momentum.py
```

담당:

- runtime JSON 읽기
- strategy 엔진 호출
- latest 결과 반환
- theme_map 상태 반환
- 승인 대기열 생성 요청을 기존 approval service로 위임

하지 않을 것:

- Seven Split 후보 스캔 수정
- 기존 `/api/candidates` 동작 변경
- 기존 `index.html` 후보 테이블 구조 변경
- 직접 KIS 주문 호출

권장 순서:

1. 전용 페이지와 전용 API를 먼저 만든다.
2. 운영 검증 후 필요하면 기존 AI 전략 후보 목록에 요약만 연결한다.
3. 자동 승인/자동 주문은 마지막 단계에서 별도 플래그로 연다.

### 6단계: 주문 승인 연동

초기 정책:

- 자동 주문 전송 금지
- `REQUIRE_APPROVAL=true` 유지
- 후보 점수 75점 이상만 승인 대기열 생성 가능
- 기존 주문 라우터/승인 대기열을 사용

승인 요청 reason 예시:

```text
내러티브 모멘텀 91.2점: AI 반도체 투자 확대, strength=88, bullish, 3일 연속 rising, 매칭 테마=반도체
```

추가 차단 조건:

- `ONLINE_ACCESS_BLOCKED=true`이면 스캔/주문 후보 생성 중 외부 호출 금지
- stale 내러티브면 승인 대기열 생성 금지
- 가격 조회 실패 시 주문 수량 계산 금지
- 관리종목/거래정지/상폐 위험 종목 필터 적용
- 같은 종목의 pending buy approval이 있으면 중복 생성 금지

### 7단계: 백테스트와 성과 검증

내러티브 전략은 과거 뉴스 JSON이 없으면 정밀 백테스트가 어렵습니다. 따라서 초기에는 다음 방식으로 검증합니다.

1. 매일 장 시작 전 내러티브 JSON을 저장한다.
2. 후보를 `scanned_candidates`에 저장한다.
3. 1일/5일/20일 forward return을 기존 후보 성과 갱신 기능으로 축적한다.
4. KOSPI/KOSDAQ, 기존 Seven Split 후보, 급락반등 후보와 성과를 비교한다.

검증 지표:

- 후보 수
- 평균 `final_score`
- 1일/5일/20일 평균 수익률
- 승률
- MDD
- 후보 중복률
- 기존 전략 대비 초과수익
- stale 또는 매핑 실패 비율
- LLM JSON 파싱 실패율

자동주문 허용 기준 예시:

```text
최근 60거래일 기준
- 후보 50개 이상
- 5일 평균 수익률 > KOSPI 5일 평균 수익률
- 5일 승률 >= 52%
- 최상위 5개 후보의 손실 꼬리 리스크가 기존 후보보다 낮음
- JSON 파싱 실패율 < 5%
```

## 8. 파라미터 초깃값

| 파라미터 | 초깃값 | 설명 |
| --- | --- | --- |
| `STRENGTH_MIN` | `70` | H1 기본 60보다 보수적으로 시작 |
| `DAY_WEIGHTS` | `[1.0, 0.6, 0.3]` | 최근 3거래일 반영 |
| `SENTIMENT bullish/positive` | `1.2` | 긍정 내러티브 가중 |
| `SENTIMENT neutral` | `1.0` | 중립 유지 |
| `SENTIMENT mixed` | `0.8` | 혼재 뉴스 감점 |
| `SENTIMENT negative/bearish` | `0.5` | 매수 후보에서는 강한 감점 |
| `ACCEL 2일` | `1.10` | 2일 연속 rising |
| `ACCEL 3일` | `1.25` | 3일 연속 rising |
| `SHIFT new` | `+15` | 신규 강한 테마 |
| `SHIFT rising` | `+5` | 강화 테마 |
| `SHIFT fading/disappeared` | `-10` | 약화 테마 |
| `THEME_MATCH_THRESHOLD` | `0.45` | 초기 fuzzy 매칭 |
| `STREAK_MATCH_THRESHOLD` | `0.55` | 같은 테마 연속성 판정 |
| `APPROVAL_SCORE_MIN` | `75` | 승인 대기열 생성 최소점 |

## 9. LLM 수집 전략

초기에는 자동 뉴스 크롤러보다 **수동/반자동 JSON 입력**을 권장합니다.

1차:

- 운영자가 당일 핵심 뉴스 10~30개를 수집한다.
- LLM으로 정해진 JSON 스키마만 생성한다.
- 사람이 JSON을 검토한 뒤 `.runtime/narrative_history.json`에 저장한다.

2차:

- KIS 뉴스, 네이버 금융 뉴스, 공시 요약을 수집한다.
- LLM 출력은 JSON schema 검증 후에만 저장한다.
- 같은 뉴스가 중복 반영되지 않도록 URL 또는 제목 해시를 기록한다.

3차:

- 장중 재계산을 허용한다.
- 단, 장중 재계산 결과는 즉시 주문하지 않고 대시보드 경고/후보 갱신으로만 사용한다.

LLM 프롬프트 원칙:

- 출력은 JSON만 허용한다.
- `strength`는 근거 뉴스 수, 기업 규모, 시장 영향도를 함께 고려하게 한다.
- `sentiment`는 주가 방향성이 아니라 해당 테마의 수급 기대 관점으로 평가한다.
- `direction`은 전일 대비 부각 정도로 판단한다.
- 모르는 종목명이나 티커는 생성하지 못하게 한다. 종목 연결은 반드시 `theme_map`에서만 한다.

## 13. 최종 권고

내러티브 모멘텀은 한스톡에 적용 가치가 있습니다. 다만 이 전략의 핵심 리스크는 매매 로직보다 **입력 데이터 품질**입니다.

따라서 구현은 다음 원칙을 지켜야 합니다.

- 점수 엔진은 결정론적으로 만든다.
- LLM은 종목 추천자가 아니라 내러티브 JSON 생성기로 제한한다.
- 종목 연결은 `theme_map`만 사용한다.
- 실거래는 검증 전까지 승인 대기열을 반드시 거친다.
- 성과 판단은 영상 사례가 아니라 한스톡의 `scanned_candidates` 이력과 forward return으로 한다.

이 방식이면 유튜브 영상의 아이디어를 한스톡 구조에 맞게 흡수하면서도, 현재 프로젝트의 안전장치와 테스트 체계를 해치지 않고 단계적으로 적용할 수 있습니다.

## 16. 스케줄 자동 실행 전면 개편

2026-06-19 기준으로 내러티브 대시보드를 수동 스캔 화면에서 스케줄 실행 가능한 운영 화면으로 확장했습니다.

핵심 변경:

- `src/strategy/narrative_momentum_runner.py`를 추가해 수동 스캔, 즉시 실행, VM 스케줄 실행이 같은 러너를 사용합니다.
- `strategy_id=narrative_momentum_strategy`가 `strategy_schedules`에 등록되면 `src/strategy_scheduler.py`가 일반 트레이더 대신 내러티브 러너를 실행합니다.
- 실행 결과는 `.runtime/narrative_momentum_latest.json`과 `scheduler_results`에 저장됩니다.
- 결과에는 `summary`가 포함되며 상태, 후보 수, 저장 수, 미매칭 수, 평균 점수, 상위 시그널을 바로 표시합니다.
- 대시보드에 `스케줄` 탭을 추가해 활성화, 간격, 시작/종료 시간, 요일, 실행 모드를 설정할 수 있습니다.
- `즉시 실행` 버튼으로 스케줄과 같은 방식의 실행 결과를 바로 만들고 이력 테이블에서 확인할 수 있습니다.
